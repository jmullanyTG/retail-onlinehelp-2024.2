Site Clustering Overview

Stores can be clustered into logical groups using Site clustering. Site clustering involves the use of:

  • partitions and statistical algorithms to cluster stores by both qualitative and quantitative criteria.
  • templates to quickly create clusters across different levels of the product and site hierarchies.
  • cluster set snapshots to evolve cluster configurations over time.

Before creating site clusters

Before creating clusters, it is vital that business discussions be had to determine what criteria are most important in addressing opportunities in the business. Also, the levels of the site hierarchy must be determined, at which the functions for which clusters are used, to facilitate efficiency throughout. As the clustering strategy is planned, consider the following:

  • How is the business structured in terms of selling channels? Is there a need to segment by full and off-priced business, by country, by geography, some combination of all? Are there other factors related to site attributes (capacity, climate, demographic, etc.) to use to group sites together?
  • How is the business structured in terms of roles and responsibilities? Are planners responsible for multiple areas of the product and site hierarchies or are they specialized to certain areas? Based on the complexity of analysis that will be expected of them going forward and the tools available, should this structure be reassessed?
  • At what level of the product hierarchy will clusters be different? Should there be homogenous clusters that are used across multiple product areas or clusters that are specific to certain areas? In those areas, are there product attributes to consider to further specialize clusters?
  • What metrics should be used when clustering stores? Typically, this involves some type of historical or forecasted sales component, but should the business use sales revenue, units, cost, margin, margin %, GMROII, some combination of multiple?

To use product and site attributes for the creation of site clusters, attribute sets, families, and values must be defined. At a minimum, at least one set must be assigned to the topmost level of both the product and site hierarchy on the Attribute Management solution on the Maintenance tab.

Site and product attribute associations can be set through configured data import processes, but they can also be managed manually in the following places:

  • Site Attributes: Management > Site Manager > Site Attribute Management
  • Product Attributes: Management > Variant Manager > Variant Attribute Maintenance

Using new cluster sets

Before beginning any activities in which the cluster sets are needed, do the following:

  1. Define the product and site hierarchies: System Utilities > Views > Maintenance > Hierarchy Filtering.
  2. (Optional) To use product and site attributes for the creation of site clusters, attribute sets, families, and values must be defined, and at a minimum, at least one set must be assigned to the topmost level of both the product and site hierarchy: Maintenance > Attribute Management.

Site and product attribute associations can be set via a configured data import process, or managed manually in the following places:

Site Attributes: Management > Site Manager > Site Attribute Management

Product Attributes: Management > Variant Manager > Variant Attribute Maintenance

Understanding site clusters

Though Site Clustering can be used to create store clusters for other purposes, its primary focus is to create clusters specifically for use in Assortment Planning.

The following table explains the concepts that underpin site clustering and how site clustering is used in Assortment Planning to facilitate planning activities.

Concept Description
Cluster Set and Clusters A group of stores is referred to as a cluster. A cluster set is typically a collection of two or more clusters that were derived using the same set of criteria, though it is possible to have a cluster set with a single cluster.

A typical cluster set in a product hierarchy has clusters with stores assigned to those clusters based on their relative sales unit volume against others stores in the set. This scenario is common where cluster sets are created by business channel and the E- commerce channel consists of only one store. Thus, only one cluster is needed in the e-commerce cluster set.
Clusters as Persistent Entities When a cluster is created in a cluster set, the cluster remains as a persistent entity in that set, regardless of whether any stores are assigned to that set. This facilitates the continuity of assortment plans, which continue on indefinitely once created, as the configuration of clusters in a set changes over time.

For a cluster set with multiple clusters, stores naturally rank into the clusters based on their sales volume when the cluster set is initially calculated. When the cluster set is recalculated with current data, stores that improved their performance rank higher which may leave a cluster with no stores. However, the empty cluster still exists because it had stores earlier in the season. In this scenario, it is possible to manually shift stores into the empty cluster, if necessary.
Cluster Sets and the Assortment Plan Site Hierarchy To be usable in a given assortment plan, a cluster set must be contained in a single portion of the lowest level of the site hierarchy assigned to that plan. This is required to allow for correct aggregation to site hierarchy levels in the various reporting and analysis functions in Assortment Planning.

An assortment plan with an assigned site hierarchy that has the lowest level of country: US, UK, and China. The business plan is divided by channel (Retail, Wholesale, and e-commerce) and product category (Apparel, Footwear, and Accessories). In that scenario, the following cluster sets are created:

  • US Retail / Apparel, US Retail / Footwear, US Retail Accessories

  • US Wholesale / Apparel, US Wholesale / Footwear, US Wholesale Accessories

  • US e-commerce / Apparel, US e-commerce / Footwear, US e-commerce / Accessories

    UK Retail / Apparel, UK Retail / Footwear, UK Retail / Accessories

  • UK Wholesale / Apparel, UK Wholesale / Footwear, UK Wholesale / Accessories

  • UK e-commerce / Apparel, UK e-commerce / Footwear, UK e-commerce / Accessories

  • CHINA Retail / Apparel, CHINA Retail / Footwear, CHINA Retail / Accessories

  • CHINA Wholesale / Apparel, CHINA Wholesale / Footwear, CHINA Wholesale / Accessories

  • CHINA e-commerce / Apparel, CHINA – e-commerce / Footwear, CHINA – e-commerce / Accessories

Because the stores in each cluster set are contained in the lowest level of the site hierarchy assigned to the assortment plan, all of these cluster sets can be used.

Using the same assortment plan as in the previous example, ignoring the lowest level of country in the plan creates the following channel level cluster sets:

  • Retail Apparel, Retail / Footwear, Retail / Accessories
  • Wholesale Apparel, Wholesale / Footwear, Wholesale / Accessories
  • e-commerce Apparel, e-commerce / Footwear, e-commerce / Accessories

Stores in the Retail and Wholesale channels exist across multiple countries (the lowest level of the assortment plan site hierarchy). Thus, these cluster sets do not pass validation by the assortment plan and cannot be used in it.

Though cluster sets and clusters are not explicitly tied to the site hierarchy, they are related to it in the manner shown by these examples. Thus cluster sets available for selection in Assortment Planning is dictated by the levels of the site hierarchy that have been selected for the analysis or view.
Cluster Set Snapshots and Event Management Cluster set snapshots save the cluster set and associated site configured cluster with an effective start date. The effective start date is used in Event Management to determine if it can be used during the time frame of the event. There must be at least a one day overlap with the event for the cluster set snapshot to be available.

Over time, site clustering needs to be modified. When a new snapshot is created with a new effective start date, an end date will be in effect for the original snapshot and Event Management will no longer make it available to events that are configured out of its time frame.

Cluster Set Snapshots and Promotion Management Cluster set snapshots save the cluster set and associated site configured cluster with an effective start date. The effective start date is used in Promotion Management to determine if it can be used during the time frame of a promotion. There must be at least one day overlap with a promotion for the cluster set snapshot to be available.

Over time, site clustering needs to be modified. When a new snapshot is created with a new effective start date, an end date is in effect for the original snapshot and it is no longer available to Promotions that are configured out of its time frame.

Cluster Set Snapshots and Assortment Planning Cluster set snapshots save the specific cluster assignments of stores within a cluster set at a point in time. Once a snapshot passes validation within an assortment plan, it is used to define not only the cluster sets that are eligible for use within that plan, but the number, specific stores, and percent contribution of the stores within each cluster.

It is possible to have multiple cluster set snapshots become effective over the course of a plan or even within the life of a product. At a minimum, one cluster set snapshot with an effective date of the earliest historical period within the plan must be set. This allows for the proper assignment of sales to clusters, which are the lowest level at which plans are managed within Assortment Planning.

Cluster set snapshots are sent to all existing assortment plans for validation as soon as they are created. Conversely, when new assortment plans are created, all existing cluster set snapshots are validated against them. No additional interaction is required to initiate the validation process.
Clustering and Qualitative Criteria Because the k-cluster type algorithms used for Site Clustering do not work with non-numeric data points, attributes with qualitative criteria cannot be specified as explicit conditions when creating cluster sets.

There are two options when clustering based on qualitative criteria:

  1. Specify the criteria as Partition By parameters during cluster set configuration. For more information, see Cluster set configuration.

    This creates separate cluster sets for each value that exists for the selected Partition By attribute, yielding the same number of total clusters as the likely desired outcome.

  2. Create and assign site attribute values to the stores to cluster that are numeric equivalents or indicators of the desired qualitative criteria. This creates a single cluster set that (potentially) ranks stores into more granular clusters, depending on the number of clusters specified.

Rather than specifying store capacity as High, Medium, and Low, specify square footage values assigned by store, or numeric minimum or median values can be assigned for each desired tier (High = 20,000, Medium = 14,000, and Small = 8,000).

There may be unpredictable results if this method is used, depending on the specific numeric values used for the qualitative criteria equivalents, the degree of separation between each, and how those interact with the quantitative data points in the set.

For example, if clustering on sales volume and average temperature, stores with high sales volume and a low average temperature could be assigned to a cluster that was intended for low to mid sales volume and high average temperature.

This method is unclear as to the specific intent of the criteria. Consequently, the system mathematically resolves the criteria for the purpose of ranking.
