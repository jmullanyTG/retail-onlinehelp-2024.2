Manipulating placeholders and collections

Grouping by columns

Placeholders and collections can be grouped by clicking and dragging any column header to the grouping section at the top of this view.

Expand or collapse groups by clicking on the arrow.

Placeholder/collection Locks

As many of the metrics within the main planning view are inter-dependent, it can be necessary to lock metrics for one or more placeholders within a collection or for the entire collection depending on what was changed until changes are saved.

In this example, the placeholder was locked because Phase In was changed.

Calculations

When checking values, it should be understood that values are presented in a rounded state for ease of display. Small differences in the actual calculated results may be present. In the database, all calculations are done with maximum precision to avoid unnecessary rounding.

checkboxes in Collection view such as those in the Active and Carryover fields display a solid gold state to indicate that only some placeholders within the collection are checked.

Changing a collection code

  1. In the PH Collection Code field, double-click or right-click and select Multiple Changes.
  2. In the dialog, select an existing code from the list and click Ok or enter a new description in the Collection Desc field and click New.

Changing a placeholder’s collection assignment recalculates both its prior Collection’s % and its newly assigned Collection’s % contribution for all placeholders that belong to either.

Changing placeholder/collection status of placeholders with replenishment data present

Changing placeholder/collection status of placeholders with replenishment data present (see Available for Replen) to Replen displays the following:

Clicking Yes irreversibly overwrites the sales plan for the changed item with available replenishment forecast data. While the change to the item Status can be reverted, the previous sales plan data is not maintained. The state of the sales plan at the time of the change, including metric values, cluster assignments, and location exceptions, become the go-forward sales plan, though the plan elements will become editable once more.

Changing placeholder/collection status of placeholders without replenishment data present

Changing placeholder/collection Status of placeholders without replenishment data present (see Available for Replen) to Replen displays the following:

Clicking Yes irreversibly zeros out the sales plan for the changed item. While the change to the item Status can be reverted, the previous sales plan data is not maintained. Plan metrics stay at zero though the plan elements are editable once more.

Notes on Replen items:

  • Sales plans for Replen items are updated nightly to coincide with updates to replenishment forecasts.

  • Replen item sales plans can also be updated on demand by clicking the Recalculate button in the Replen section of the toolbar. However, this utilizes the settings of the most recent replenishment run.

    It is advisable to verify the output of the last replenishment run created using the same parameters as are used in the nightly run or the data brought into AP may be inconsistent or incomplete.

  • As only future data is ever written in a replenishment run and data is written daily, this requires the application to specially handle inter-period sales and on-hand inventory. BOP on-hand is created from actuals and not changed throughout the period. Sales and receipts are combined as "future periods from replenishment" and "past periods from actuals" to maintain a total period number throughout the period until it is fully actualized. EOP on-hand values always come from replenishment data.
  • Certain plan elements still remain editable. Most are informational fields (PH Description and Code, Tags, Notes) but all Shrink/Inventory Adjustments metrics in Sales Plan View are editable.
  • Replen items cannot be locked, made inactive, or deleted.
  • Seasonality Curve, Size Curve, and Min Pres are cleared as these elements are controlled within the Forecasts module. If the status changes from Replen to something else, the item inherits the defaults set on the plan for the product level to which it belongs.
  • Any Sales Plan and Receipt Plan overrides are irrevocably cleared.

  • Receipt Plan constraints are cleared as these elements are controlled within the Forecasts module. If the status changes from Replen to something else the item inherits the defaults set on the plan for the product level to which it belongs.

    Currently, Forecasts uses buckets and Assortment Planning uses periods. These are two different time entity types that do not have to align in order for Assortment Planning to read in replenishment forecast data.

    The application attempts to match the calendars in each module by cross referencing Forecast bucket start dates with the Assortment Planning periods beginning and ending dates. However, if the buckets and periods do not coincide, this may lead to sub-optimal data within Assortment Planning.

    For example, if the buckets in Forecasts are set to monthly and the periods in AP are weekly, AP puts the sales for the entire monthly bucket into the first weekly period available during that month, etc.

    In Collection view, displays the most advanced status of any placeholders within the collection (with Open being least and Replen being most advanced).

Visibility to actual ROS and how aligned sales plans are with actual sales

The ROS metric reflects a combination of actual and planned sales in a single ROS metric. The following metrics split actual and planned sales out. This provides better visibility to the actual ROS for active sites and how aligned sales plans are with actual sales.

  • Reg Ftr Site Count, Reg Ftr ROS ($), Reg Ftr ROS (U), Reg Ftr Site ($), Reg Ftr Site (U), Reg Ftr AUR ($), Act ROS ($), Act ROS (U)
  • SPV ROS ($), SPV ROS (U), SPV Sales ($), SPV Sales (U)

If at any point a user makes a change causing Total Sales Units to change, this will cascade down to the Regular Future figures as well.

ROS Rate of Sale

Supersessions in AP: Using an existing variant's EOP as the BOP for a subsequent variant

Supersessions let users forecast demand for new items with no sales history by using an existing variant's EOP as the BOP for the new item.

Supersession relationships are defined in the Supersession solution at the Item Master level (style-color-size) and best practice is to map all sizes to create a 1-to-1 relationship.

If all sizes are not mapped, then the defined size curve for the PH is used to approximate the anticipated remaining units by size, for use by the supersession stock metrics.

For multi-size variants, Supersessions are defined by size, for each size. New Supersessions defined in the Supersession solution and changes to existing Supersessions are automatically pushed into AP.

Typically, outside of the Supersession functionality AP is at the variant level (style-color) and does not consider size.

When not all sizes are mapped

For example, if a style is projected to end phase-out with 2,000 units left on hand, and only 3 of the 4 sizes are mapped with a Supersession (S,M,L were mapped to a Supersession style, but XL was not) and the size curve for that PH was defined as 30-40-20-10:

The curve is applied against the remaining 2,000 units:

  • 2,000*0.3 = 600 S
  • 2,000*0.4 = 800 M
  • 2,000*0.2 = 400 L

meaning 600+800+400 = 1,800 units would be displayed under Supersession Stock Units for the upcoming Supersession style.

The remaining 200 XL units (2,000*0.1) do not transfer to the new Supersession style, as XL was not mapped.

Similarly, if all Sites are not mapped, historical site contribution is used to exclude the stock projected to be on-hand at unmapped sites.

Example of a Supersession scenario

There are times when a new variant is close enough to an existing variant that the two variants are seen by customers as interchangeable. (For example, variant A = red socks with black squares, variant B = red socks with black circles.)

In such a scenario, the stock on-hand position of the first can be considered when planning stock needs for the second variant. This is very useful when:

  • both items look very similar

  • variant B’s lifecycle is after variant A’s lifecycle and the user would like to use variant A’s EOP as variant B’s BOP

    Phase dates need to be considered when utilizing supersessions.

  • the supplier of an existing variant produces the same style each year, but changes the style-code/SKU in subsequent years
  • when stock from one item should be considered towards stock for a subsequent item.

Metrics

  • % Locked: Indicates a change has been made to the corresponding % contribution that has not yet been saved (not present in Collection view).

  • Active: Whether the current placeholder or collection is active in the assortment plan or not. Defaults to checked. If unchecked, the line is formatted with a strike-through and zeros out its Collection % Cont.

    All aspects of the placeholder/collection sales plan are retained for seamless switching between active and inactive states. Inactive placeholders/collections are not included in any reporting within Assortment Planning: Target Creation, Assortment Rollup, Reconciliation, and Dashboards.

    Placeholders/collections can also be made active or inactive. Select Set Active or Set Inactive from the right-click menu. Placeholders/collections of Replen Status cannot be made Inactive.

  • Actual First Sale Period: The period where the first sale actualized.

  • Actual First Sale: Read-only and displays in date format. This field only populates after the actualization of an initial sale. If the number of days to actualize does not reach as far back as the Phase In date, then the value displayed in this field may not be accurate.

    This field will not populate if the placeholder is no longer being updated with actualized values.

    The value in this field depends on the value set for existing system policy Assortment days history to actualize.

    Actual Sales U: The sum of all actualized sales units (from historical periods) for the placeholder/collection.

    Actualization occurs as part of overnight processing (exact frequency can vary by implementation) in which the sales units that were realized for a given period are fed to the ToolsGroup JustEnough database and read into Assortment Planning. Planned sales values are overwritten by these actual sales values.

  • Actualized Periods: The number of actualized periods present in the life of the placeholder/collection.

    For a placeholder whose Phase In was Wk 1 and it is now Wk 5 and assuming regular overnight processing has occurred, then the actualized periods is 4.

  • Actualized: Indicates the presence of any actualized periods during the placeholder/collection’s life.

  • Additional Buy: The quantity of additional non-sales and non-min pres units to buy for the placeholder or collection. The primary function of this field is to show discrepancies between the sum of Ttl Sls U and (Min Pres * Site Count) and Ttl Buy U that can occur because of the ability to directly edit S/T %.

    To remove unwanted excess units and rebalance a placeholder or collection’s sales plan, zero this field out and save.

    Calculation: (Ttl Buy U – Ttl Sls U – (Min Pres * Site Count))

  • Alloc Status: Average retail (selling) price excluding VAT per unit for the placeholder item

  • AUR ex VAT: Average retail (selling) price excluding VAT per unit for the placeholder item

    Calculation: AUR / (1 + VAT %)

  • AUR: The average unit retail price for the placeholder/collection. Because of the potential for multiple currencies, VAT (value added tax) rates, and promotional participation to vary by site within a single cluster, AUR is not directly editable and is only changed as a result of changes to Home AUT, Phase In, Phase Out, and promotion applications.

    Calculation: (Ttl Sls $ / Ttl Sls U)

  • Home AUT: The average unit ticket price for the placeholder/collection in the Home Site currency. Displays the weighted average of AUT values for all future periods beneath. Home AUT is assumed to be inclusive of the Home Site’s VAT and is treated as such for all dependent calculations.

    Because of the potential for multiple currencies, VAT (value added tax) rates, and promotional participation to vary by site within a single cluster, changes to Home AUT require saving to properly calculate the effects on the rest of the plan for a given placeholder/collection..

    Period level changes to Home AUT can be made directly in Sales Plan view.

    • Blended AUT (Blended Average Unit Ticket) and AUR (average unit retail), being an outcome of the calculation mentioned above, is not directly editable.

    If Import Config setting AUT for planned periods is set to 2 (AUT is only managed through imports), then Home AUT is read-only for Placeholders with a status of Assigned, Committed, Replen, and Allocate. For placeholders with a status of Open and Created, Home AUT is editable.

    Regarding scenarios where Import Config settings AUT for planned periods and Receipt AUC for Planned Periods are set to 3 (Home AUT is managed through both imports and the UI, with a "last in wins" principle), data is only imported if the Last Modified Date on the variant cost/price data is later that the date logged against the placeholder.

    If an import sets Home AUT to 12, and then the user changes the value to 13, the next run will not reset the value back to 12. If price data is updated via imports to 14, then the next maintenance window will set Home AUT to 14.

    Additionally, periods will only have an imported Home AUT or Receipt AUC value set if that value actually exists in price/cost data. It will not set all missing periods to zero.

  • Blended AUT: The average unit ticket price for the placeholder/collection, accounting for all price conversions across different currencies and countries within the cluster. Displays the weighted average of Blended AUT values for all periods, both actualized and future, beneath. Blended AUT is derived from Home AUT in future periods, but it comes from provided sales history for actualized periods. Blended AUT is assumed to be inclusive of VATs for all sites within the cluster and is treated as such for all dependent calculations.

    Blended AUT incorporates each of the following, all of which are individually controlled via system policy:

    • VAT rate differences between the Home Site and sites within each cluster
    • Price Exception factors
    • Price Matrix conversions

  • Available for Replen: A read-only checkbox indicating whether replenishment forecast data is available for the placeholder or collection.

    It is important to see if this field is checked before changing the status of a placeholder or collection to Replen. There is a warning if no data is present. However, checking first saves time. This is also useful for verifying that the replenishment forecast matches up with the assigned variants.

    This field is checked if ANY portion of the combination assigned variants, cluster sets, and product lifespan has a replenishment forecast against it. For example, the field is checked if a single item master related to the assigned variant has a forecast for a single site contained within the clusters that variant is assorted to during the lifespan of the variant.

    Forecasts must be set for all intended item masters and sites to ensure completeness of replenishment data in Assortment Planning.

  • Carryover: Whether a placeholder/collection is an already existing item from a previous window of time. This is driven solely off a placeholder status of Committed. Committed placeholders/collections have confirmed on order quantities for actual variants and cannot be edited to the same degree as placeholders/collections that have yet to be ordered.

  • Cluster Sets: The cluster sets the placeholder or collection assorts to as set in the Cluster Details side pane.

  • Collection % Cont: Editable percentage (10.1%) The contribution percentage of a placeholder to the collection to which it belongs (not present in Collection view). Placeholders in collections by themselves always have a percentage of 100%. When shifting placeholder collection assignments, percentages for all placeholders in affected collections are recalculated, but sales hold for all. When adjusting Collection % Cont directly, collection level sales hold and placeholder sales recalculate.

    As changes are made to this metric within a collection, the % Locked field is checked, so subsequent changes won’t alter previous entries within the collection. The sum of Collection % Cont for all placeholders in a collection must equal 100%. Entries are re-normalized that would cause the sum to exceed by dividing each by the sum of the percentages.

    Calculation: Ttl Sls U / (Ttl Sls U for all placeholders in collection).

    Entering values above 100% generates an Invalid Value error.

  • Color Code: (not present in Collection view). The color code for the placeholder. When a variant is assigned it overwrites any user-entered value and displays the variant color code. On unassigning a variant, reverts back to the user-entered value, if any.

  • Color: (not present in Collection view). The color description for the placeholder. When a variant is assigned it overwrites any user-entered value and displays the variant color description. On unassigning a variant, it reverts back to any user-entered value.

  • Curve Mode: The mode of the selected Seasonality Curve:

    Mode Description
    Static The selected curve is applied exactly as it exists at the time of application. The placeholder or collection’s curve will not change if changes are made to the curve, whether manually in the Curves module or through dynamic recalculation. Think of this as making a copy of the applied curve.
    Dynamic Changes made to the curve are pushed down to the placeholder/collection and the dependent sales plans recalculate when those changes are pushed.

  • Default Lead Days: Either the default lead time in days set for the Product Level the placeholder is assigned to or the default lead time of the supplier associated with the variant assigned to the placeholder.

    If a variant is assigned to the placeholder but that variant has no associated supplier, the default lead time in days set as the plan default is used. For more information on setting default lead days, see Tracking Against Ordering Deadlines.

  • External Exists?: Whether external data exists for the given variant in the database.

  • External Status: Whether Clean or Dirty.

  • Health Error Details: Click the dropdown arrow to display additional details regarding messages displayed in the Health Error field.

    Example of Cluster Sets with zero site counts

  • Health Error: An error message indicating if there is an inconsistency or issue with an element of the placeholder’s sales plan. More details about the particular cluster sets, clusters, or periods are included in the Health Error Details column. For more information on the error messages and how to resolve errors, see Health Error Codes.

    The Health Error and Health Error Details columns can help identity placeholders where inconsistencies have occurred. Issues brought to light in these fields should be rectified as soon as possible to prevent further data drift from occurring.

  • Image: (not present in Collection view) A thumbnail of the primary image for the placeholder or attached variant. If Image Tool tip is enabled in View Options, then hovering over the thumbnail image displays a larger version of the image.

    Images can be assigned to placeholders prior to variants being attached, but are replaced by the actual variant images once a variant is attached. To retain placeholder images when attaching variants, go to Variant Manager > Image Maintenance to attach and add the placeholder images there, or add them after the variant has been attached.

    Where a Product Lifecycle Management (PLM) system integration is present, the primary image of a placeholder is not editable once a variant has been assigned. This is because the image is dictated via data imported from the PLM. The ability to edit the primary image is controlled by system policy Variant Primary Image Classification Update (Module: Assortment).

    For more information, see Image mode.

    To edit, right-click the image and choose:

    • Edit Primary to display the Image Maintenance dialog allowing selection of another primary image.
    • Edit All to display a dialog allowing editing and deleting of existing images or addition of more images. This can also be launched by double-clicking the image thumbnail.

  • Last Actualized Date:

  • Last Actualize Failed:

  • Last Alloc Push: Updates on successful completion of a push to Allocations.

  • Last External Pull: The last time data was pulled or recalculated.

  • Last Replen: The date replenishment data was last updated for the placeholder/collection (valid for placeholders/collections of Replen status).

  • Last Retrend Date: The date the Retrend operation was last performed.

  • Lifecycle Curve: The life cycle curve applied to the placeholder/collection. Displays (External) to indicate the Curve in use is from External data. (External) will only show if the Sales Units metric is set to non-editable in the Manage Import Config view (the External Editability field is set to No).

  • Location Exceptions: Indicates the presence of location level exceptions for the placeholder or collection. This is checked if any sites contained in clusters the placeholder or collection is assorted to have been excluded via the Manage Locations function.

    Location exceptions do not alter Site Count. This is because when location exceptions are made the placeholder or collection ROS recalculates according to each site’s percent contribution to the cluster. Also, decrementing Site Count causes the resulting sales plan to drop too low as doing so essentially double-counts the removal of the excluded sites.

  • Lock Type: Displays the lock type if any part of a placeholder/collection line is locked.

  • Locked: Locks or unlocks the collection the placeholder belongs to (or the collection in Collection view) for editing. When checked, all placeholders attached to the collection are locked and read-only in the view and highlighted in pink or red depending on the row band.

    Actualization of sales in a period can still affect locked placeholders/collections. Creation of new cluster set snapshots for cluster sets to which locked collections are assorted can also change locked placeholders/collections. This is done to maintain synchronicity with ROS metrics and the site count associated with clusters to which locked placeholders/collections are assorted.

    Notes:

    • Recalculation of linked Seasonality Curves does not affect locked placeholders/collections.
    • Already active promotions remain active on locked placeholders/collections.
    • New promotions cannot be applied to locked placeholders
    • Placeholders/collections can also be locked by right clicking and selecting Lock Selected or Unlock Selected.

  • Margin %: The gross margin percentage for the placeholder or collection. An editable value that adjusts the Receipt AUC metric to allow this margin to be achieved. This allows Planners to hand over Assortment Plans to their Buyers together with the information about a desired or target AUC they should be aiming for when negotiating prices with suppliers. Coloring rules follow the Receipt AUC pattern.

    Calculation: ((AUR ex VAT / (1 + VAT %)) – AUC) / (AUR ex VAT/ 1 + VAT %)

  • Markdown Curve: The curve applied to the placeholder or collection. Used to break down Ttl Sls $ and Ttl Sls U by period in Sales Plan View. For more information, see Sales Plan View.

    Displays (External) to indicate the Curve in use is from External data. (External) will only show if the Sales Units metric is set to non-editable in the Manage Import Config view (the External Editability field is set to No).

    Only curves of the Manual or Specified Conditions Using Enhanced Filtering types are available for selection within Assortment Planning.

    Click the ellipsis () to launch the Curve Selection screen. For more information, see Curve Selection Screen.

    Within the Assortment Planning module, multiple curves can be selected to calculate a Sales Plan. These curves trend toward zero when a markdown is later added to a placeholder or variant. An individual markdown curve can be specified to be applied separately after the markdown date.

    Whether Period Match or Scale To Fit is used to match the markdown curve to the sales plan is determined by system policy Algorithm To Match Markdown Curve to Sales Plan. Though not required, it is recommended to choose a curve that trends downward over time. System policy Markdown Curve Default Behavior controls what happens to a markdown sales plan if no markdown curve is selected. Depending on how Markdown Curve Default Behavior is configured, either a line trending to zero is used to decrease the sales plan to zero at phase out or the sales plan does not slope to zero and the markdown sales plan is left unchanged.

  • Min Pres: The desired minimum presentation stock per site for the placeholder/collection. Min Pres represents additional non-sales units to be purchased for the placeholder or collection to maintain adequate stock for presentation within each store.

    Calculation: SUM (cluster site count * cluster min pres – for all clusters) / Site Count (yields a weighted average)

  • Mkdn Date: Selects the markdown date for the placeholder/collection. Selections are limited to periods from Phase In through Phase Out. Mkdn Date sets which period to stop respecting the various constraints utilized in the Receipt Plan supporting view including Min Pres.

    When viewing the by-period plan for the placeholder in Sales Plan or Receipt Plan view, Min Pres drops to zero in the period specified by Mkdn Date. An existing markdown date splits the plan into pre- and post-markdown data. Promotions can be applied to markdown periods.

    • If no markdown date has been specified, then the plan exists as standard.
    • If a markdown date is added at time of the placeholder creation, then indexes for both halves of the plan are normalized together, and total sales is spread across.
    • If a markdown date is added after the total sales has been entered, then sales plan pre-markdown is locked in as it already exists, and the markdown plan is re-angled to head to zero.

    A markdown date can be pushed by changing Phase In or Phase Out. If Phase Out changes, the zero-trend point moves with the phase out and the slope of the markdown curve changes accordingly.

    For information regarding extending markdown dates, see Extending the markdown date in Placeholders.

  • Next Order Date: The earliest order deadline for receipts planned for the corresponding placeholder based on assigned default lead times. Essentially, this is the earliest receipt date for any cluster level receipt planned for the placeholder minus Default Lead Days.

    This should also correspond with the earliest Order By Date shown in Receipt Plan view for the placeholder.

  • No. Periods: The MAX period count for the placeholder/collection. Because Phase In and Phase Out at the placeholder/collection level are set to the earliest and latest periods, respectively, of all clusters to which it has been assorted, No. Periods must calculate this way to balance the other calculations in the main planning view.

    A placeholder is assorted to two clusters: cluster A has Phase In/Phase Out of Wk 1 and Wk 4 and cluster B has a Phase In/Phase Out of Wk 2 and Wk 5.

    The placeholder Phase In/Phase Out is Wk 1 and Wk 5, yielding a No. Periods of 5.

  • Notes: An informational field that supports greater than 256 characters.

  • OOT Sales: Whether the item meets or exceeds the product hierarchy level default threshold for sales units outside an assorted cluster. A mark indicates the threshold has been met or exceeded.

    The OOT Sales field indicates that there have been noteworthy sales of units at non-assorted sites. This helps provide a total picture of sales including those sales outside of the assorted cluster for a placeholder or variant. All decisions regarding what to do once the threshold is reached or exceeded are user driven and not system driven.

    For example, users may or may not decide to include the site in a cluster set moving forward. If a site was excluded from a cluster definition last week but is included in this week’s snapshot, then last week’s sales and stock are considered as being to a non-assorted site, but this week’s would be to an assorted site.

    Use the Assortment Out-of-Tier Report supporting view to drill down further into the data. For more information, Assortment Out-of-Tier Report View.

  • OOT Stock: Whether the item meets or exceeds the threshold at the product hierarchy level for the number of units in stock at sites that were not originally assorted to. A mark indicates the threshold has been met or exceeded.

    The threshold is at the placeholder level, dependent on selected cluster filters. Therefore, if the stock on hand product hierarchy threshold for a placeholder is set to 500 units, and there are two cluster sets, A and B, each of which had 250 units stock on hand, then:

    • The checkbox is marked if only cluster A or only cluster B are selected.
    • If both clusters are selected, the checkbox is marked as the combined selected cluster sets meet the threshold.

  • Origin Date: When the origin was created.

  • Origin Details: The details of the original variant that was used as the basis for the placeholder when created by Smart Start or Hindsight. This column is empty if the placeholder was not created by either Smart Start or Hindsight. If created via Smart Start, then details include the scenario. If created via Hindsight, then details include the placeholder it came from.

    The Origin fields are blank if the placeholder was created prior to implementation of this software version.

  • Origin Type: The type of origin. For example, Smart Start or Hindsight.

  • Origin User: Who created the origin.

  • Pack Size: (not present in Collection view) The total number of pieces in a pack for every size curve that is assigned to a placeholder. This field is only populated for Placeholders with a defined Packing Instruction.

    For example, for a size curve for younger boys outerwear tank tops, if there are 6 different sizes and there is 1 piece of each size in a pack, then the value that displays is 6.

    The actual value is the Pack Size selected in the in the Pack Instruction Selector in the Packing Instruction supporting view. Changing the selected Pack Size updates the metric in the main planning view after Execute is clicked and the change confirmed.

    For more information on the Packing Instruction supporting view, see Packing Instruction view.

  • PH Code: (not present in Collection view) A unique system generated code for the placeholder.

  • PH Coll Desc: A description for the collection. Changing this for one placeholder changes it for all other placeholders currently assigned to that collection.

  • PH Collection Code: The collection to which the placeholder is assigned (read-only in Collection view). Change a single PH Collection Code by double-clicking in the field. Right-click and select Multiple Changes to change multiple codes simultaneously.

  • PH Count: (not present in Placeholder view). The number of the placeholders assigned to a collection.

  • PH Desc: (not present in Collection view) A description for the placeholder. Placeholders generated through the Smart Start process default to Smart Started from Segment and (segment name) for easy identification.

  • Phase In: The phase in date for the placeholder or collection. If set, selections are limited to periods prior to Phase Out.

    Defaults to the earliest non-actualized period within the Window of Time in the data selection parameters

  • Phase Out: The phase out date for the placeholder or collection. If set, selections are limited to periods after Phase In. Defaults to the last period within the Window of Time in the data selection parameters.

    For placeholders/collections of Replen Status, Phase Out dates are derived from the maximum Phase Out date set in the Forecasting module for all relevant inventories.

  • Planned Phase In: Editable with the date chosen by clicking in the field and selecting a date from a calendar.

  • PLM Code: (not present in Collection view). If integrated with a product lifecycle management (PLM) system, then this field displays the PLM code associated with the assigned variant.

    If the View Option Active Style Bank has been selected, then the PLM Code field acts as a dropdown and shows the same Variants as the Style Bank view. For more information, see Advanced Style Bank.

  • PLM Desc: If integrated with a product life cycle management (PLM) system, then this displays the PLM description associated with the assigned variant (not present in Collection view).

  • Product Level: The level of the product hierarchy to which the placeholder/collection is assigned. Recalculation of linked Seasonality Curves does not affect locked placeholders/collections. This selection does not have to be made at the lowest level of the product hierarchy, though it is recommended users do so to avoid ambiguity in reporting.

  • Promo: Indicates whether there are any active promotions applied to the placeholder/collection. Checked means at least one promotion is applied to the placeholder/collection. Unchecked means no promotions are applied. Inactive promotions will not cause the box to be checked.

    Specific promotions applied can be viewed in the Promotions bottom pane view. For more information, see Promotions view (not available in Collection view).

  • Receipt AUC: The average unit cost price for the placeholder/collection. Displays the weighted average of AUC values for all periods, both actualized and future, beneath.

    Once a placeholder or collection has begun to actualize (current date is greater than Phase In date), AUC is read-only.

  • Receipt Status: The calculation status of the receipt plan for the placeholder/collection (Clean or Dirty).

    • In the main planning and Cluster Sets views, changes to Seasonality Curve, Collection % Contribution, Phase In, Mkdn Date, Phase Out, Home AUT, Ttl Sls $, Ttl Sls U, ROS $, ROS U, Min Pres, Ttl Buy $, Ttl Buy U, Additional Buy, or S/T will set this to Dirty.
    • In Sales Plan view, changes to Selling Period, Sales, Sales Units, Shrink Units, Shrink Units % of Sales Units, Shrink Cost, Shrink Cost % of COGS, Inventory Adjustment Units and Inventory Adjustment Cost will set this to Dirty.
    • In Receipt Plan view, changes to Periods of Supply, Periods Before Phase In, Min Stock to Sales, Shrink % of Sales, Minimum Presentation, Receipt Period, and Receipt Units will set this to Dirty.

  • Reg S/T %: Read only. The calculation is Reg SPV Sales U / Ttl Orders U.

  • Replen Status: The Replen Status (Clean or Dirty) indicating whether the placeholder or collection has been updated with the most recent Replenishment forecast data (valid for placeholders/collections of Replen status only).

  • Retrend The fields for Retrend metrics populate only after a user selects the Retrend option in the Process Queue section of the main planning view toolbar. The following display the results and status of the Retrend operation:

    • Retrend ROS ($)

    • Retrend ROS (U)

    • Retrend Ftr ROS ($)

    • Retrend Ftr ROS (U)

    • Retrend Status (showing a status of either Clean or Dirty)

    An additional metric, Last Retrend Date, displays the date the Retrend operation was last performed.

    There are Notifications for Retrend submission status in the Inbox/process queue .

  • Reviewed?: Allows users to mark that a placeholder has been reviewed and that commitments are ready to be generated.

    If the field is checked, then all underlying Cluster Sets have been reviewed. Unchecked indicates that none have been reviewed and multi-select indicates a mix of reviewed and non-reviewed at the Cluster Set level.

    To edit, users require the User Right Allow the ability to mark Placeholder as Reviewed.

    This field is visible only when system policy Require Placeholder Review before Commitment Generation is set to True.

  • ROS $: user editable currency ($10.01) The rate of sale (average sales per site per period) in revenue for the placeholder/collection.

    Calculation: (Ttl Sls $ / No. Periods / Site Count)

  • ROS U: The rate of sale (average sales per site per period) in units for the placeholder/collection.

    Calculation: (Ttl Sls U / No. Periods / Site Count)

  • S/T % Delta: Compares the targeted sell through rate against the actual. The calculation is (Target S/T % - S/T %) / (S/T %).

  • S/T %: read-only. Percentage (10.1%). Compares the lifetime sales for the item against the lifetime orders.

    Calculation: (SPV Sls U / Ttl Orders U)

  • Seasonality Curve: Click the ellipsis (...) to launch the Curve Selection screen. For more information, see Curve Selection screen.

    Displays (External) to indicate the Curve in use is from External data. (External) will only show if the Sales Units metric is set to non-editable in the Manage Import Config view (the External Editability field is set to No).

  • Seasonality Curve Factor: When an Item has both a Seasonality Curve and a Lifecycle Curve, the Seasonality Curve Factor determines how the two curves influence the Sales Plan. A value of .5 means that the two curves each influence the Sales Plan 50% each. A value of .7 would mean that the Seasonality Curve is weighted at 70% and the Lifecycle Curve at 30%. The values for the two curves always add up to 100%. The Seasonality Curve Factor is only relevant if there is both a Seasonality Curve and a Lifecycle Curve. If only one curve is set, then any value in the Seasonality Curve Factor field does nothing.

  • Site Count (Active): The average total site count and the average active site count (in parenthesis) of all clusters a placeholder/collection is assorted to between Phase In and Phase Out.

    Typically, this metric is always a static whole number that equates to the sum of the cluster level site counts to which the placeholder/collection is assorted. However, when multiple cluster set snapshots are present during a placeholder or collection’s life, it is possible for site count by cluster to vary, thus necessitating the need for an average count.

    Size Curve: The size curve to apply to the placeholder or collection. Displays all defined size curves. Filter by typing in the Size Curve Option or Description fields. Use % to do a “Contains” filter condition rather than “Starts With” (for example, %Tops returns everything containing the word Tops).

    Applying at the collection level pushes a size curve down to all placeholders in a collection. The relevance of sizes contained in the applied curve to new placeholders cannot be validated.

  • SPV Overrides: Indicates if overrides to Sales $ or Sales U have been made to the placeholder or collection in the Sales Plan supporting view

    Changes made in Sales Plan view do not directly affect the metrics in the main planning view with the exception of removing Selling Periods. These metrics are considered the System Plan when working in Sales Plan view. This lets users see differences in the higher level placeholder and cluster plans created in the main planning view and the results of more granular period level adjustments made in Sales Plan view.

  • SPV Diff ($), SPV Diff ($) %, SPV Diff (U), SPV Diff (U) %: The difference between SPV Sales and Total Sales.

  • SPV Sales ($): The sum of all sales for the placeholder/collection in Sales Plan view. Provided for easy comparison to the plan at the placeholder/collection level.

  • SPV Sales (U): The sum of all sales Units for the placeholder/collection in Sales Plan view. Provided for easy comparison to the plan at the placeholder/collection level.

  • Status: The status for the placeholder collection (read-only in Collection view). In Collection mode, Allocation is the second most advanced Status, behind Replen.

    • Replen: A selectable for placeholders that are either in Assigned or Committed Status. This shifts control of the placeholder/collection sales plan to replenishment forecasts in the Forecasts module making the plan read-only within Assortment Planning.

    • Allocation: A single directional push to Allocations with the variant data being pushed by the allocation strategy status. Plan editability in AP is the same as committed items.

      Allocation Strategy StatusData that pushes from AP to allocation strategies
      New ItemVariant cluster rate of sales units, phase in & out, markdown date, min pres, size curve, cluster assignments, and location exceptions.
      Ongoing ItemOnly Phase Out and Markdown Date are pushed, with all other aspects of the sales plan being managed within Allocations.
      ReplenishmentNo data pushes.

    • Committed: Auto-assigned upon confirmation of units on-order for the placeholder.

      As actual units on-order is always against a variant (real style-color), once a placeholder enters Committed status, the assigned variant cannot be removed nor can the status be changed unless the on-order units are canceled.

    • Assigned: Auto-assigned upon assignment of a variant to the placeholder. Assigned placeholders inherit the Variant ID, PLM Code, PLM Description, Color Code, Color Description, and Attributes of the assigned variant. For more information, see Attributes View.

      Placeholders whose variants are unassigned via Undo > Unassign Selected are automatically returned to Created status.

    • Created: Denotes placeholders that cannot be deleted by Smart Starting and overlapping Target Creation scenario.
    • Open: Default for all new placeholders. Denotes placeholders that can be deleted by Smart Starting an overlapping Target Creation scenario. For more information, see Target Creation.

    • External: (Does not display for placeholders in Open or Created status.) Not part of the workflow, this is a status that certain Variants can be assigned to mark them as different from regular Placeholders.

      A Confirmation warning message displays if changing to External status and the placeholder contains no external data. In this case, selecting Yes to proceed irreversibly zeros out the sales plan.

  • Supplier: A variant's default supplier. The field shows a combination of Supplier Code and Supplier name. While the placeholder has not been assigned to a variant, this field is editable letting users define a Supplier for a placeholder before the placeholder is assigned to a variant. The dropdown includes Supplier Code and Supplier Name as searchable columns.

    This field updates to the variant's default Supplier on assignment. The Style Bank view also has a Supplier field which represents the variant's default Supplier.

    If system policy Enforce Supplier matching before Variant Assignment is set to True, then the assignment of a variant to a placeholder can only occur if the Suppliers chosen are the same. If View Option Active Style Bank is selected, then only records with matching supplier records are shown.

  • Tags: Allows the ad-hoc creation and assignment of informational descriptors for a given item.

    Multiple tags can be entered for a single line with inputs being confirmed by the use of the comma ( , ) key. As users begin to type, any tags already present in the current view that begin with the entered characters appear in an auto-complete selector below the field. These entries can be selected via click or by using the arrow keys to navigate to the desired item and the comma key to select.

    Target S/T %: Editable, has no calculation effect or relationship. The user can enter their target sell through value to compare against the calculated S/T %.

  • Total Orders ($): The sum of Total Orders Cost from Receipt Plan view for the placeholder/collection. This is included as a comparison of the actual receipt plan to the estimated Ttl Buy $ in the main planning view.

  • Total Orders (U): The sum of Total Orders from Receipt Plan view for the placeholder/collection. This is included as a comparison of the actual receipt plan to the estimated Ttl Buy U in the main planning view.

  • Ttl Buy ($): The total buy in revenue for the placeholder or collection. Changes to this metric directly affect the placeholder or collection sales plan.

    Calculation: (Ttl Sls U + (Min Pres * Site Count) + Additional Buy) * AUC

  • Ttl Buy (U): The total buy in revenue for the placeholder/collection. Changes to this metric directly affect the placeholder or collection sales plan.

    Calculation: (Ttl Sls U + (Min Pres * Site Count) + Additional Buy)

  • Ttl Sls ($): Sets the total sales revenue for the placeholder/collection from Phase In through Phase Out

    Calculation: (ROS $ * No. Periods * Site Count

  • Ttl Sls (U): Sets the total sales units for the placeholder/collection from Phase In through Phase Out

    Calculation: (ROS U * No. Periods * Site Count)

  • Variant ID: (not present in Collection view). A unique system identifier for the assigned variant. This field is blank if no variant is assigned.
