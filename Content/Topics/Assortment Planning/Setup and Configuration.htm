Setup and configuration

Before creating assortment plans, the following setup and configuration must be completed:

  • Definition of product and site hierarchies: System Utilities > Views > Maintenance > Hierarchy Filtering.
  • Optional: To use product and site attributes for the creation of site clusters, Attribute Sets, Families, and Values must be defined. Also, at least one set must be assigned to the topmost level of both the product and site hierarchy: Maintenance > Attribute Management. Site and product attribute associations can be set via configured data import processes and also be managed manually in the following places:
    • Site Attributes: Management > Site Manager > Site Attribute Management
    • Product Attributes: Management > Variant Manager > Variant Attribute Maintenance

  • Optional: If managing the sales plans of ongoing items through the Forecasts module (replenishment forecast integration), homogenizing the dates is recommended for the time buckets used in forecasting and the periods to be used in Assortment Planning. Review the information for Replen Status items to understand the specifics of how such management works between the two modules.

    For more information, see Status.

  • Creation, snapshotting, and validation of cluster sets to used: (Management > Sites > Site Clustering):

    Cluster set validations occur automatically on plan creation and when cluster sets are snapshotted.

  • Definition of Currency Exchange Rates, VAT Rates, and Price Translations:

    DefinitionDescription
    CurrencySystem Utilities > Views > Currency > Currency Exchange Rates
    VATSystem Utilities > Views > Currency > Planning VAT Rate
    Price TranslationsSystem Utilities > Views > Currency > Price Translation Management and Price Translation Hierarchy Assignment

    Users not requiring VAT or Price Translations functionality must still set a rate/factor of 1 (indicating no change) and assign it for all site/product hierarchy levels, as these values are utilized in many system calculations within Assortment Planning.

  • Set foundational system policies:

    • The following are related to foundational data handling within Assortment Planning. For a complete listing of Assortment Planning system policies, see Assortment planning related system policies.

    System Policy Description
    Assortment days history to actualize The number of days back to go when actualizing sales, receipts and stock on hand for assortment planning. The default is 60.
    Assortment days history to maintain The number of days back to maintain detail data for in a live assortment plan. Data older than this date is summarized and then deleted. Entering a value of zero keeps all history. The default is 800.

    This policy is useful for database size management. However, the amount of historical data maintained has a direct correlation with the data that can be used to seed historical placeholder/variants in the Seed process. Also, the date from which to begin seeding cannot exceed the date designated by the current date minus the value set for this policy. For more information on the Seed process, see Seed.

    Days to keep deleted data The maximum number of days that deleted data is held (hidden) in the database.  Set a lower number for more aggressive deleting and more space conservation. The default is 30.
    Default Commitment Merge Sets the default merging behavior of order lines when commitment orders are created: 0 = None, 1 = Merge by Variant (default). For more information, see Creating Commitment Orders.see Commitment Orders.
    Receipt Cluster Merge All When commitments are generated, whether to group receipts for variants together across clusters by default, or whether to keep them separate by default.
