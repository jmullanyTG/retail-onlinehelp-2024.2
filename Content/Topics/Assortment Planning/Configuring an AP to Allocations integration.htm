Pushing data from Assortment plans to Allocation strategies

Users working within both Allocations and Assortment Planning can push Assortment plan data to Allocations. This significantly improves the alignment of Assortment and Allocation strategies within business teams.

Determine a link items' eligibility

The main entity used to link a placeholder and allocation strategy line is the Variant entity. A placeholder and strategy line with the same variant can be linked with the following conditions.

Allocation Strategy requirements

  • the strategy must already exist as strategies are not created by AP.

  • be assigned a variant

    In Allocations, multiple strategies can exist for a single variant, even though this is not standard behavior. If this is the case, AP pushes assortment data to all variant strategies, duplicating data.

  • is a Level type strategy line, not a Group type

  • is Active

  • is not Replen type

Placeholder requirements

  • be in Committed status

  • have Phase In in the future

Linking placeholders and strategies

In the Setting Source field in the Allocation Strategy grid, select Assortment Planning.

Once Setting Source is set to Assortment Planning and all other eligibility requirements are met, then the checkbox in the Alloc Strategy field in the main planning view displays a check mark. This indicates a valid strategy exists that can be linked to the placeholder. In the Status field, set the placeholder status to Allocation. Upon saving, this establishes a formal link between the placeholder and strategy.

Allocation status considerations:

  • Placeholders in Allocation status behave the same way as in Committed Status. All fields are still editable before phase in, and some fields are editable after phase in.

  • Filters can be created for items in Allocation status.

  • A collection’s status is shown as the most advanced status for all placeholders in the collection. Replen is the most advanced followed by Allocation, Committed, Assigned, Created, Open. For example, if a collection has placeholders in each status, the collection status is Replen. If it has placeholders in all statuses but Replen, then the status is Allocation, and so on.

  • The status can be changed to Allocation even if the Alloc Strategy column is not checked. However, this operation is unnecessary as no Allocation Strategy exists.

Unlinking placeholders and strategies

Allocators can unlink strategies from AP pushes by setting Setting Source back to Allocation. This disables pushes for the matched placeholder.

Pushing AP data to linked strategy lines in Allocations

AP data is pushed to linked strategy lines by selecting the Push Allocations option under Alloc in the Process Queue dropdown menu. The Process Queue button is found in the toolbar of the main planning view.

The data push will queue for placeholders in Allocation status currently visible in the main planning view at the time the push is initiated, but will include all data for those variants, whether or not it is currently shown.

This means that if a variant is assorted to more than what is visible on the main planning view (i.e. other cluster sets or other placeholders mapped to the same variant excluded by current filters), that data will also push to the allocation strategy at the same time.

This breaches security for user rights, since it is possible that users without access to certain product/site levels will have the ability to push that data to Allocation. For example, a user only has access to the USA cluster set, and so only sees the placeholder’s USA assortment, but the variant is assorted to the CAN cluster set as well. When the USA user clicks Push, all data, including data for the Canada cluster sets, is pushed to the strategy. A message displays indicating this case if it arises.

After a successful push, the Last Alloc Push column displays the last push date and time. The Notifications module notifies the user who initiated the push, as well as the user who last modified the strategy.

Differences in ROS values between AP and Allocations

ROS values may not exactly match between AP and Allocations, because each module uses the main denominators - site count and period count - slightly differently.

Cluster set / cluster definitions are defined by Snapshots in both modules.  However, in AP, the snapshot in use is the valid snapshot effective during each period of the life of the item. Which site belongs to which cluster set / cluster can change over time.  In Allocations, the snapshot in use is the one effective during the phase-in date of the strategy, meaning sites belong to only one cluster set / cluster. 

This means that AP pushes ROS values for a blended cluster set / cluster definition across the life of the item, but Allocations uses that ROS to determine demand for the cluster set / cluster as defined at phase in, which could lead to under or over allocating.  If this is not desired, the phase periods must be changed to match the effective cluster set snapshots.

Site count

Closed sites

In AP, a closed site is not included in site counts.  In Allocations, it is included in initial display site counts, and further demand and allocation projection engines exclude closed sites during their processing.

Site exclusions

When a site is excluded in AP, ROS is scaled down using the % contribution of the excluded site, and this is the value displayed in AP.  However, the displayed ROS value in the grid on the Allocate Cluster Rate of Sale tab in Allocations does not include the site exclusion factor. It is the ROS as if all sites were participating.  This is because the forecast mechanism takes site exclusions into account in its processing and scales the output appropriately.

Period count

AP uses calendars based on periods, and Allocations uses buckets. If the two calendar systems are completely synchronized, then there should be no discrepancy in the period count divisor for ROS. However, if there are differences, those differences are reflected in changed ROS values.
