Managing assortment plans and setting plan defaults

To access the Manage Assortment Plans screen, from the Plan tab, click Assortment Planning and then Manage.

The following functionality is available on the Manage Assortment Plans screen:

New

Add a new assortment plan. For information describing the buttons and fields, see General Tab .

Edit

Launches the Update Assortment dialog for the selected assortment plan.

Note that most of the plan parameters are now locked.

User Access tab: User and role permissions for the assortment plan are edited in the same manner as described in the User Access section below.

Delete

Deletes the selected assortment plan.

User Access

Launches the Manage Access view for the selected plan. Access can be set at the User level or by Role with roles being assigned as users are added to the system.

The Product and Site Hierarchy access levels available is dependent on the corresponding parameter selections for the assortment plan.

The Select All and Select None buttons in the Product and Site fields in the upper portion of the view can be used to quickly apply or remove complete access to all available Product or Site hierarchy levels within the plan without having to interact with the bottom portion of the dialog.

Buttons

Button Description
Save Saves permissions changes made to the assortment plan.
Cancel Closes the Manage Access view without saving changes made to the assortment plan.
Save and Close Saves permissions changes made to the assortment plan and closes the Manage Access view.

User permissions for assortment plans can also be edited via the User Access tab of the Update Assortment dialog (see Edit section above). Additional permissions can also be setup in the user manager.

Snapshot

Launches the Manage Snapshots dialog in which users can plan snapshots which occur on a regular schedule and to create and delete snapshots manually. Snapshots are captures of the state of the current plan at the time they are generated for the purpose of future comparisons as users work with the plan. Snapshots can be used within the Sales Plan, Assortment Rollup, and Reconciliation views within Assortment planning.

For details on the metrics included in snapshots, see Comparison metrics listed in under the respective sections for these supporting views.

For descriptions of buttons and fields, see Snapshot Tab .

Defaults

Launches the Manage Assortment Defaults view for the current plan. Users can set default parameters for plan elements and functions used frequently within Assortment Planning so they do not have to be manually set each time. Default parameters can generally still be changed later by the user if needed.

Defaults set here are applied to placeholders assigned to the specified level of the product hierarchy at the time of generation, whether manually within the main planning view or through the Smart Start function.

Defaults are set in an inheritance scheme, meaning defaults set at higher levels of the product hierarchy persist down through the levels beneath unless different defaults are set at those lower levels as exceptions.

Changing defaults in this screen does not alter settings for placeholders that already exist.

A hierarchy has a department called Mens Tops. Beneath Mens Tops there are the sub-departments Mens Tees, Mens Wovens, and Mens Knits. A user sets a default at the Mens Tops level and then sets a different default at the Mens Knits level. Mens Tees and Wovens inherit the same default that was set at the Mens Tops level, but Men’ Knits uses the default set at its own level. All default settings are not required.

For more information on the tabs in the default screen, refer to the following:

  • Assortment Plan Defaults tab
  • Assortment Plan Defaults By Cluster tab
  • Assortment Plan Defaults By Time tab

Receipt Periods

Launches the Manage Receipt Periods view in which Receipt periods can be defined in a pivot table. The Manage Receipt Periods view provides detailed visibility to how receipts are planned and provides a way to effectively manage Receipts at an aggregated hierarchy level with Receipt periods assigned to specific weeks.

Receipt periods can be assigned at any level of the Product hierarchy and can be assigned to all underlying levels. Exceptions can be made on underlying levels in the hierarchy. Only those time periods selected using the Window of Time parameter populate the grid.

The Receipt periods are applied to all existing Placeholders whose phase dates intersect with the selected time periods, and the defaults will be applied to all new placeholders created in the future.

Defaults

The following defaults are available within each field:

Default Description
Flow Candidate A possible flow period. Whether the period is a flow period is determined by the Periods Before Phase In and the Flow Period Frequency.
Flow Date Specifies that the selected period is a flow date. If selected, then the period will always be opted in as a receipt period.
No Flow Date Specifies that the selected period is not a flow date. If specified, then the period will never have flows going into it for the items in the product level for the Cluster Set Clusters.

The default setting is determined by the system policy Default Flow Date Type.

Setting a default for all sub-categories of a hierarchy level

A default can be assigned at a higher level (for example, Coats & Jackets) and all lower levels are automatically set to that default. This saves having to configure many sub-categories.

For example, to set Flow Date as the default for all sub-categories of Coats & Jackets for a particular set of weeks, click on the Accessories label to highlight all the relevant fields. Within the grid, right-click and select the Multiple Changes menu option to update multiple fields using the Multi Data Change Form. This updates all the sub-categories simultaneously.

Alternatively, a particular hierarchy level can be collapsed by clicking on the arrow and set the default in the appropriate field in the grid.

Selections available by period

Once saved, the field selections act as defaults. This means anything created after the default was set inherits the default.

Existing items are not affected unless either Apply Visible, Saved Defaults to Existing Items or Apply Visible, Saved Defaults Over User Overrides is clicked.

Applying changes to existing items

The user can apply all Receipt Period defaults to all existing items by clicking one of the two buttons on the top of the screen. For both, users must click Save before the defaults on screen are applied. If the user does not click Save before clicking an Apply button, then the unsaved changes will not be in effect.

The changes brought on by clicking these buttons only affect items in any of the hierarchy levels and cluster set / clusters currently displayed in the grid.

If there are placeholders that are assigned to a parent hierarchy level but haven’t been assigned a final product level, then all placeholders assigned at that parent level are affected. For example, placeholders assigned to Mens > Accessories that have not been given a more specific assignment (Bags, Belts, etc) will be affected if the Mens > Accessories level is on the screen.

The date selection parameter determines which placeholders are affected in terms of how their Phase dates overlap. If any point of the life cycle of the placeholder intersects the selected Window of Time, then all flow dates for the planned life of that placeholder are updated, not just the periods being selected.

Changes are applied to existing items in one of two ways:

Option Description
Apply Visible, Saved Defaults to Existing Items Applies the defaults on screen to all existing items, but leaves user overrides made in Receipt Plan intact. It only changes flows made by previous applications of Receipt Periods default data or automatically generated flow dates.
Apply Visible, Saved Defaults Over User Overrides Applies the defaults on screen to all existing items, including any user overrides made in Receipt Plan. There is a confirmation message to ensure the user intends to clear out all placeholder-specific Receipt Plans.

Mixed defaults within an aggregate level

If there are mixed defaults within a hierarchy level, then when the field is collapsed the formatting changes. The text is removed and the field is highlighted yellow to indicate there are different states below. Conversely, if all defaults below are the same the formatting does not change.

Collapsed view

Seed

Launches the Seed Placeholders dialog. Because of the way placeholder entities are used within Assortment Planning, a necessary step in creating new plans is generating equivalent placeholder entities for existing items for which historical selling information is to be utilized.

Ideally, this process is performed once per plan as all new items created in Assortment Planning are generated as placeholder entities. However, the process can be re-run to pick up new items that weren’t present at the time of seeding without adversely affecting anything that has already been seeded. The process looks for variants that don’t have placeholder entities created and creates them.

Two system policies allow for the specification of custom user-friendly descriptions for placeholders and collections generated from the Seed process. A style description or variant description is still respectively appended at the end of the custom description as an indicator to the seed link (for example, ‘custom text from system policy’ + style or variant description). The two system policies are:

  • Description for Generated Placeholder Collections from Seed
  • Description for Generated Placeholders from Seed

Seeding and Attribute Inheritance

When seeding placeholders from historical variants, any attributes attached to those variants will also be attached to the resultant placeholders, provided the following are true:

  • A valid attribute set is associated with the product level to which the variants belong. Association can be by directly choosing an attribute set for that level of the product hierarchy or through inheritance of a set from a level above.
  • That attribute set is assigned to both the Variant and Variant Placeholder entity types.
  • That attribute set includes all attribute families containing the attribute values associated with the variants placeholders are being created for.
  • All applicable attribute families are assigned to both the Variant and Variant Placeholder entity types.
  • All attribute values associated with the variants placeholders are being created for are marked as Allowed for the corresponding attribute set.

For more information on creating and assigning attribute sets, families, and values, see Attribute Management and Assignment Overview.

For more information, refer to the Attribute Management online help.

Seed Placeholders parameters

Start Date: Sets the historical date the system begins at when generating placeholders for existing items. This should be set to the farthest date back desired to utilize historical sales in Assortment Planning. A general best practice is at least 2 years as most businesses typically view data this far back as being relevant to current plans.

Start date cannot fall before the date designated by the current date minus the number of days specified in the Assortment days history to maintain system policy (accessed via Maintenance > Policies > Module: Assortment).

Seed Placeholders grid

Use Default Cluster Set(s) per product level: This option appears at the top of the cluster set list and allows users to easily seed historical placeholders for the default cluster sets assigned by product level. For this option to work properly, at least one default cluster set must be assigned either directly or through inheritance from a level above for all product levels present in the plan being seeded. An error message displays indicating the need to do this if it has not been done at the time seeding is initiated.

This option cannot be used in tandem with the selection of other cluster sets within the list. An error displays if there is an attempt to do so.

Due to the dynamic nature of cluster configurations, all that is needed here is to select the cluster sets that contain the sites where the items sold historically. Sales assignments are made at site level and returned based on the active cluster set snapshot for a given period.

Split Factors

Split factors are managed using the Manage Split Factors screen within the Assortment Planning module. For more information, see Managing Split Factors.

Grid columns

Cluster Set: A read-only description for each validated cluster set within the current plan.

Overlapping Cluster Sets: A read-only description for any validated cluster sets that overlap with a given cluster set (it contains some or all of the same sites as another set). While it is possible to have overlapping cluster sets within a single plan, it is not possible to assort placeholder items to multiple overlapping cluster sets. One cluster set per group of sites must be chosen in which to plan.

Current Placeholder Count: A read-only count of all placeholder items currently generated that registered sales in a given cluster set from the Start Date forward.

Buttons

Refresh: Refreshes the Manage Assortment Plans dialog.

Execute: Selects the cluster sets in which the historical items to generate placeholders for were present.

Fields: Opens the fields Customization dialog. Users can drag fields to/from the Assortment Seed Placeholders column header area and this dialog to customize the view. Custom layouts can be saved/restored by right-clicking in the Assortment Seed Placeholders area and selecting Save Layout or Restore Layout, respectively.

Save: Initiates the placeholder generation procedure for the selected cluster sets from the Start Date forward.

Cancel: Unchecks selected cluster sets in the Assortment Seed Placeholders section or closes the Manage Assortment Plans view without opening an assortment plan.

Ok: Opens the selected assortment plan and launches the Smart Start screen.

Import Config

The configuration of sources of external data is done using the Manage Import Config screen. To display this view, click the Import Config button on the Manage Assortment Plans toolbar. The screen displays a list of AP metrics and allows for the configuration of each source of external data in one place.

Data Source Type Configuration pane

To choose the Source Type for each field, click in the field to display a dropdown list of options. The options that display depends on whether a price (AUT/AUR) or a cost (AUC) type is selected.

Metric notes

The Seasonality Curve, Lifecycle Curve, and Markdown Curve fields in the main planning view display (External) to indicate the Curve in use is from External data. (External) will only show if the Sales Units metric is set to non-editable in the Manage Import Config view (the External Editability field is set to No).

The metric Projected Sales Units is configurable. Depending on how this metric is configured, the Projected Sales Units line in the Sales Plan view populates from either Replen or External.
