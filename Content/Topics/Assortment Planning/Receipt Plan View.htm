Receipt Plan view

The ideal receipt strategy for items to be purchased can be identified using the Receipt Plan view. Functionality includes the ability to:

  • set constraints, such as minimum Periods of Supply

  • define the periods in which receipts are allowed to arrive and how many receipts are needed

    This can be overridden in the Receipt Periods view. The Receipt Periods view provides an aggregate level view for specifying if particular periods are or are not flow dates for all items in a particular level of the product hierarchy. See Receipt Periods.

  • send commitment orders to Order Manager for final processing, review, approval, and submission to external systems.

Toolbar functionality

Parameters: Displays or hides the parameters pane at the top of the view.

Parameter Description
View Mode The view mode for receipt plan view:
  • Total Company: the revenue and unit metrics as total company values – meaning the total volume for all sites contained in whatever level the Pivot Grid is aggregated to
  • Average Store: the revenue and unit metrics as average store values – meaning the per site volume for the number of sites contained in whatever level the Pivot Grid is aggregated to.

Execute: Refreshes the view. This is not typically necessary, as the view refreshes on its own when a new placeholder or collection is selected in the main planning view, when changes are applied, and when the view mode changes.

Other options

  • Save: Saves all changes
  • Cancel: Cancels all changes and reverts them to their previously saved values
  • Export: Exports and saves the pivot grid in many common formats including Microsoft Excel and Adobe PDF
  • Open: Opens the pivot grid in many common formats including Microsoft Excel and Adobe PDF and does not require a save prior to viewing
  • Graph: Graphs the selected data in the view

  • Fields: Allows the user to customize the field layout for the view. Custom layouts can be saved and restored by right-clicking in the Pivot Grid and selecting Save Layout or Restore Layout, respectively.

System policies

A number of system policies affect the output of the Receipt Planning Engine. These system policies are managed in Maintenance > Policies. However, this should not be done without the assistance of a technical consultant. Possible exceptions are:

Assortment Engine DC-to-Cluster Lead Time: This setting determines the Default Lead Time Days from the DC to the Cluster that will be used to calculate Next Order Date information. This will be rounded to the nearest period.

When the engine is calculating receipts, this setting essentially determines the range of periods it must consider to meet the defined constraints for a given period. See examples for Periods of Supply and Min Stk to Sales below for details.

Assortment Engine Supplier-to-DC Lead Time: Sets the Default Lead Time Days from the Supplier to the DC that is used to determine eligible receipt dates. This will be rounded to the nearest period. Default value is 7.

Allow Receipts After Markdown: When set to False (default), the Receipt Plan only considers the regular portion of the Sales plan and ignores the markdown phase of the plan. Set the system policy to True if additional stock should be ordered beyond the planned markdown date.

If system policy Editibility of Receipt AUC is set to 2 (AUC is only managed through imports), then AUC is read-only for Placeholders with a status of Assigned, Committed, Replen, and Allocate. For placeholders with a status of Open and Created, AUC is editable.

Fractional values in quantity fields, recalculation, and rounding

If a fractional value is entered in a quantity field (in any way), the recalculation step does the following:

  • values below 1 and above 0 round to 1, otherwise standard rounding rules apply.

    In the UI, values appear to round to 0 but after the calculation (when the SQL Server pivot recalculates) such values round up to 1.

  • for values entered at an aggregated level, a rounded split ensures that all values are whole numbers and that the total is split (for example, a value of 41 entered at the top level splits to 11 - 10 - 10 - 10).

Modifying receipt planning constraints for multiple Placeholders

Multiple Placeholders can be selected in the main planning view and their receipt planning constraints modified in the Receipt Plan view. However, if multiple Placeholders are selected, then the Assortment Receipt Plan section in the Receipt Plan view is greyed out.

After editing the constraints for multiple Placeholders, create Commitments by clicking Create Commitments. For more information, see Creating commitments.

To restore default constraints, select the individual Placeholders one at a time in the main planning view.

Recalculating receipts for multiple Placeholders with a Receipt Status of Dirty

To recalculate multiple Placeholders with a Receipt Status of Dirty:

  1. In the main planning view, select multiple Placeholders with a Receipt Status of Dirty.

  2. In the Receipt Plan view, click Recalculate Receipts.

    The Receipt Status for the selected Placeholders updates to Clean.

Recalculating Receipt plans without Dirty status

Users can click the Recalculate Receipts button when selected on a single placeholder and rerun the receipt engine, even if the status is not Dirty.

Editable metrics

Denoted by light blue row coloration, these are the only metrics that are editable in Receipt Plan view. All other metrics are calculations or reads from other sources.

Changing Minimum Presentation in this view updates Min Pres in the Main Planning and Cluster Set views, and in Sales Plan View.

Changing Receipt Units in this view does not update metrics in the Main Planning or Cluster Set views. This is by design. Receipts are suggested in Assortment Planning via a complex algorithm considering sales and all user-specified constraints. As sales are considered in the output already, to update Ttl Buy U would cause the sales plan to change. This might potentially invalidate the results of the receipt calculation. However, it updates Receipt Units in Sales Plan view, as will changing Receipt AUC.

Calculations

Receipt plans are calculated via the Receipt Planning Engine, which uses a complex algorithm to determine the optimal receipt flow based on sales and minimum presentation by period and user-defined constraints. As a result, system receipt recommendations require a user action to recalculate rather than recalculating dynamically as changes are made in the user interface.

Recalculation can be done either:

  • for the selected placeholder or collection only by clicking Recalculate Receipts in this view, or
  • for all placeholders/collections that require recalculation by clicking Recalculate in the main planning view header area.

Receipt Planning Engine outputs

The System Receipt Units and Ideal EOP Units metrics are outputs of the Receipt Planning Engine, which must be re-run when any of the inputs it considered change. These fields are conditionally formatted any time a change is made that could invalidate the results of the last engine run. Changes to sales, min pres, shrink, inventory adjustments, or receipt constraints all fall into this category.

Plan Horizon versus Receipt Planning Engine Horizon

If the Plan Horizon, set at the time of plan creation, extends beyond the receipt planning engine horizon for a given item, data that extends past the receipt engine horizon appears in blue italics with zero values. See example in Plan Horizon vs. Replen Forecast Horizon below.

Understanding receipt engine recommendations

The following metrics surface diagnostic information in the Receipt Plan view to better explain why the receipt engine recommended the quantities it did.

Metric Description
System Demand The demand signal, derived from Sales Units, that the engine uses when considering a receipt plan. It is inclusive of constraints, like lead time.
System Unconstrained Demand The demand signal, derived from Sales Units, that the engine uses to generate the System Demand series. It does not incorporate constraints.
System Inbound Units The aggregation of all metrics that indicate what stock is already on hand or will shortly be available. It includes On Order data, stock in transit, receipts already delivered in the current period, user-specified receipts for future periods, and the portions of DC stock allocated to the cluster.
System Lost Sales The lost sales quantity as calculated by the receipt process.

Placeholders and collections in Replen status

When a placeholder or collection is in Replen status, all relevant data is pulled to form its sales plan from the Forecasts module. Users should be aware of the following differences in this view when dealing with Replen items:

Read Only: All normally editable metrics in Receipt Plan view become read-only being populated directly from aggregated replenishment forecast data.

Receipt Constraints: All receipt related constraints are managed within the Forecasts module. Thus, receipt constraints in Assortment Planning for placeholders and collections with a status of Replen are cleared. With a change of status to something other than Replen, the default constraints assigned to any corresponding product level is reinstated.

Receipt Periods: For placeholders and collections with a status of Replen, the necessary orders to support forecasted sales are generated from the Forecasts module and sent to Order Manager for processing. Users do not need to set receipt periods in this view the way they do for non-Replen items. Receipt Periods are set automatically based on when the generated orders are scheduled to arrive.

  • Periods of Supply: Sets the periods of supply (period type as defined by the current assortment plan) for the placeholder or collection as a minimum stock constraint for receipt engine outputs. Actual by period constraint is defined as the sales planned for this many periods forward from a given period. This setting interacts with the system policy Assortment Engine Lead Time.

    Assortment Engine Lead Time is set to 7 days and Periods of Supply is set to 2. A receipt is to arrive in week 1.

    A placeholder with a Phase In date of wk 4 and a Periods Before Phase In of 2. The default calculated initial receipt for the placeholder is in wk 2.

    A placeholder with a Phase In date of wk 3, a Periods Before Phase In of 2, and a Flow Cadence of 4.

    3 receipts will occur in wk 1, wk 7, and wk 11. This is because Periods Before Phase In ensures product can be in stores by the Phase In date. Thus, the first flow date is considered to be the Phase In date for the purposes of receipt engine calculation.

  • Min Stk/Sales (Minimum Stock to Sales): Sets the minimum stock to sales ratio for the placeholder or collection as a minimum stock constraint for receipt engine outputs. Actual by period constraint is defined as the sales planned for a given period multiplied by the constraint value (period sales of 10 * Min Stk/Sls of 4 = EOP stock requirement of 40).

  • Shrink % of Sales: Sets the anticipated shrink units per period as a percentage of planned period sales units for the placeholder or collection as a minimum stock constraint for receipt engine outputs.

    A period with planned sales of 500 units and a Shrink % of Sales of 2.00% must have an EOP units of at least 10 for this constraint to be satisfied for the period.

  • Min Stock to Sales: Editable to 2 decimal places where it can be changed. The minimum stock to sales ratio as a constraint for the receipt engine outputs.
  • Target S/T %: Editable, has no calculation effect or relationship. The user can enter their target sell through value to compare against the calculated S/T %.
  • S/T %: read-only. Percentage (10.1%). Compares the lifetime sales for the item against the lifetime orders. Calculation: (SPV Sls U / Ttl Orders U).
  • Recalc Receipts: Sets whether System Receipt Units and Ideal EOP Units recalculate for the placeholder or collection when Recalculate Receipts in Receipt Plan view or Recalculate in the main planning view is clicked.
  • Hold Receipts: Whether the current total Receipt Units for the placeholder or collection can change to accommodate Placeholder Cluster Constraints upon recalculation. If unchecked, the system calculates the receipt strategy to meet the specified constraints for as long as possible without increasing Receipt Units by moving receipt units forward in time.

    If unchecked, there will be scenarios where the system cannot satisfy user defined constraints for the life of the item within the cluster.

  • Min Pres Multiplier: Multiplies the minimum presentation units quantity passed into the receipt engine by the value entered in this field for the number of weeks specified in Min Pres Multiplier No of Periods.

    Min Pres Multiplier and Min Pres Multiplier No of Periods together have an effect on the receipt plan by weighting stock received toward the beginning of the receipt plan according to the values set.

    Min Pres Multiplier is set at 2.5 and Min Pres Multiplier No of Periods is set to 5. If the minimum presentation is 15, then the minimum presentation is raised to 37.5 for the first 5 weeks.

  • Min Pres Multiplier No of Periods: The number of periods that the multiplier is in effect from the phase-in date. A value of zero means effectively there is no multiplier. Once the number of periods has expired the minimum presentation values return to the original value.

    As the item starts actualizing, the number of periods declines until it is zero. Therefore, if the multiplier is to be used over the first 3 periods of a receipt plan, then:

    • Before phase in, wk1, wk2, and wk3 all use the multiplier.
    • When wk1 is actualized and the current week is wk2, then wk2 and wk3 will use the multiplier.
    • When wk2 is actualized and the current week is wk3, only wk3 will use the multiplier.
    • When it is wk4 or later, the multiplier is no longer used, even though 3 periods continues to be the Min Pres Multiplier No of Periods value set.
  • New Site Min Pres Multiplier: This value is applied against the minimum presentation for only the site count with new store openings.
  • New Site Multiplier Number of Periods: Applies only to the site count of new store openings.

Assortment Receipt Plan grid

Metrics that account for stock in transit

In Assortment Planning, Total Order metrics are inclusive of supplier orders and internal stock movements (also known as Inter-Branch Transfers or IBT). The Assortment Receipt Plan grid has metrics that differentiate between inbound supplier orders and IBT orders providing visibility to these components.

Additionally, On Order metrics provide visibility to supplier orders that have been sent to the supplier (Committed), received back from the supplier as confirmed (Confirmed), or if there's an unknown status (Unknown).

The Past Due metrics provide visibility to orders that were expected to be receipted in the recent past. Past Due is broken out to provide visibility to all orders that are past due as well as current. These metrics provide visibility to where stock is, whether it's late or on time, whether it's moving inside the channel, etc.

These metrics are available in some or all of the following: Receipt Plan view, Sales Plan view, Assortment Rollup view, and the Reconciliation Report.

Accounting for stock in transit within clusters

The Stock In Transit to Clusters Units and Stock In Transit to Clusters Cost metrics allow BOP to be better understood by actual stock on hand components and stock in transit components.

Stock In Transit to Cluster is stock in transit to stores within a cluster specifically whereas Total Stock In Transit is stock in transit to any locale in the network, store or DC). Metric Stock In Transit to DCs Split is the difference between the two.

Committed On Order Units

Units for which commitment orders have been created but which do not have corresponding confirmed purchase orders. Auto-populated when commitments are created for periods having Receipt Units. This fields reads from commitment orders for the corresponding variant/cluster/period in Order Manager. Receipt units for a given period are transitioned out of Receipt Units to Committed Units when commitments are created for that period.

Commitment Units can be automatically decremented based on the presence of confirmed On Order Units but only if the Import Config setting Order Units for planned periods is set to Variant Orders and ToolsGroup JustEnough has integrated with an ERP/POM system capable of returning order data with the UUIDs ToolsGroup JustEnough generates for each order sent. If this is not the case, a custom procedure is needed to appropriately decrement Commitment Units and avoid duplication of units with On Order (Unknown) Units (where purchase order units the system is unable to tie to specific commitment orders populate).

Confirmed On Order Units

Confirmed units on order for the period. Calculation: (purchase order units not yet received but scheduled for the period)

When the Import Config setting Order Units for planned periods is set to Variant Orders and ToolsGroup JustEnough has integrated with an ERP/POM system capable of returning order data with the UUIDs ToolsGroup JustEnough generates for each order sent, this field populates with confirmed purchase order quantities by variant/cluster/period and decrement Committed Units accordingly. Where the system policy is set to Inventory Orders and/or external systems are NOT able to return UUIDs by order, purchase order units will appear in the On Order (Unknown) field.

On Order Units (Unknown)

Confirmed units on order for the period which cannot be matched to specific variant/cluster commitment orders. Calculation: (unlinked purchase order units not yet received but scheduled for the period)

On Order (Unknown) units split across applicable clusters using the same splitting logic as discussed in EOP Units.

    Ensuring future recommended orders are sufficient for ideal EOP quantities

    If an override receipt value exists in a period, that period becomes ineligible for system recommended receipts and system receipt units is zero.

    EOP Units of the last actualized period

    For placeholders/collections for which inventory exists in a DC, the EOP Units of the last actualized period includes a portion of available DC on-hand split out per the following logic:

    Calculation: (Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units) / SUM(Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units for all clusters) * DC on-hand units

    While there is an inherent degree of potential inaccuracy in splitting DC on-hand inventory via this method, this was a conscious design decision to assume optimal distribution of available inventory based on future demand by cluster, as the alternative of including complete supply chain logic would have had a significant impact on the performance of the system.

    Calculation: System Receipt Units – Committed Units – On Order Units – On Order (Unknown) Units

    Once a commitment order has been created, the quantity that was in this field moves into the Committed Units field indicating that an order has been generated for the planned quantity. As the order moves through the commitment process, the units will transition or On Order Units (or On Order (Unknown) Units) and then ultimately show again as receipt units when the period in question actualizes, provided the units were actually received in that period. This ensures that units for the same order are not double-counted as they move through the process.

The additional quantity can be entered and Create Commitments clicked. This can happen automatically as the sales plan changes as a result of receipt plan recalculations. Each individual order contributes to the totals shown and is tracked and transitioned through the applicable fields individually.

The structure of the receipt plan, including flow dates and % to total of receipt units by period, is respected while the system balances the period level entries to match the new Grand Total.

    Calculation: From Sales Plan view

  • Site Count: Count of all sites the placeholder or collection is assorted to at whatever level the pivot grid is aggregated to for the period. This number can vary by period if multiple cluster set snapshots are effective during the life of the placeholder or collection.

    The site count does not necessarily tell if new sites opened or closed during the period. For example, changes may be due to sites moving in or out of a cluster or, if there are two periods where the site count is consistent, there might be an opening and closing in the same week. To verify, review site information in the Site Manager and Site Clustering solutions.

  • Stock Units to Sales Units: Ratio expressing the relationship of BOP Units to Sales Units.

    Calculation: (BOP Units / Sales Units)

  • System Receipt Units: The system recommended receipt units for the period.

    When set to False (default), system policy Allow Receipts After Markdown prevents the system from ordering additional stock past the planned markdown date. The default is False. Set the system policy to True if additional stock should be ordered beyond the planned markdown date.

    Calculation: Receipt Planning Engine

    This metric seeds System Receipt Units in Sales Plan view as well.

  • Supplier Order Cost: External on order - cost of orders from suppliers not yet received. The calculation is: Committed + Confirmed + Unknown Units On Order (Total Orders - Inter-Branch Transfers)

  • Supplier Order Units: External on order - orders from suppliers not yet received. The calculation is: Committed + Confirmed + Unknown Units On Order (Total Orders - Inter-Branch Transfers)

  • Total Orders AUC: Average cost per unit of inbound orders for the period.

    Calculation: (Total Orders Cost / Total Orders Units)

  • Total Orders Cost: Cost of inbound orders for the period.

    Calculation: (Total Orders Units * Total Orders AUC)

  • Total Orders Units: Sum of all planned and confirmed incoming orders for the period.

    Calculation: (Receipt Units + Committed Units + On Order Units + On Order (Unknown) Units)
