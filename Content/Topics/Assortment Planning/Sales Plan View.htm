Sales Plan view

Period level edits to sales plans of selected placeholders or collections can be made using the Sales Plan view. Edits can be either in total or by average store and compared to the high level plan created in the main planning view of the Placeholder or Collection views. Inventory can be tracked and shrink and adjustments planned for more detailed control both pre and in-season.

Making changes to plans in the Sales Plan view

Matching Sales Plans with Life Cycle and Seasonality Curves

Where the periods between a Sales Plan and a Life Cycle or Seasonality Curve do not exactly match, configure the system to either match up the periods of the Sales Plan with the periods of the Life Cycle or Seasonality Curve or stretch or shrink the curve to match the size of the sales plan.

Which behavior is used is determined by system policies Algorithm To Match Seasonality Curve to Sales Plan and Algorithm To Match Lifecycle Curve to Sales Plan. For more information, on setting these system policies, see Assortment planning related system policies.

Viewing actualized internal stock movement metrics

Four metrics provide further visibility to internal stock movements (also known as Inter-Branch Transfers or IBT) between locations for historical periods. These metrics are only populated for actualized periods with future periods always being zero or null.

These metrics are included in the calculation to provide the current period’s BOP value with all stock being considered regardless of where it is.

The metrics are:

  • Actual Stock Units In Transit Cluster

  • Actual Stock Units In Transit DC Split

  • Actual Stock Cost In Transit Cluster

  • Actual Stock Cost In Transit DC Split

Metric naming convention

  • % +/-: Percent difference between the relevant metrics.

  • +/-: Additive or subtractive variance between the relevant metrics.

  • Calculations: Calculations in Sales Plan view all work based on one-directional locking calculations, depending on the metrics changed.

    For more information on the calculation model used in this view, see the supplemental Microsoft Excel file Assortment Planning – Sales Plan View.

  • Comparison metrics: Denoted by purple row coloration and (Comparison) appended to the end of the metric. For example, Sales (Comparison). These metrics are direct data reads from the selected Comparison Snapshot and default to zero if no snapshot is selected or no data is present in the snapshot.
  • Editable metrics: Denoted by light blue row coloration. These are the only metrics that are editable by the user in Sales Plan view. All others are calculations or reads from other sources:
    • * System, ex VAT, and Comparison metrics are never editable in Sales Plan view

      • With the exception of Selling Period, edits made in Sales Plan view DO NOT change the plan values in the main planning view and Cluster Sets view. This is by design to allow for comparison of original plans with period level changes. However, the Sales Plan view Sls $ and Sls U metrics can be brought into the main planning view to view these edits at the placeholder or collection or cluster set/cluster level.

  • Ex VAT metrics: Denoted by ex VAT appended to the end of the metric name (for example, System Sales ex VAT). Where VAT (value added tax) is in play, ex VAT metrics provide accurate profit calculations (margin) after taxes have been removed.

    Where VAT is not in play, no customization of calculations is necessary. Navigate to System Utilities > Views > Maintenance > Planning VAT Rate and set the VAT rates for all levels of the product hierarchy assigned to the assortment plan to 0. This sets the ex VAT metrics equal to the non ex VAT metrics.

  • Past Due: Orders that were expected to be receipted in the recent past

  • Plan Horizon vs. Receipt Planning Engine Horizon: If the Plan Horizon, set at the time of plan creation, extends beyond the receipt planning engine horizon for a given item, data that extends past the receipt engine horizon appears in blue italics with zero values (for example, $0.00).

    See example in Plan Horizon vs. Replen Forecast Horizon below.

  • Plan Horizon vs. Replen Forecast Horizon: If the Plan Horizon, set at the time of plan creation, extends beyond the replenishment forecast horizon for a given item, the values for the periods that go beyond the replenishment forecast are formatted in blue and italicized. These periods also display zeros for the metrics being read from replenishment data, as no forecast exists:

  • Replen Placeholders/Collections: When a placeholder/collection is of status Replen, all relevant data is pulled in from the Forecasts module to form its sales plan.

    Users should be aware of the following differences in this view when dealing with Replen items: Read Only: all normally editable metrics in Sales Plan view become read only, being populated directly from aggregated replenishment forecast data, with the exception of all Shrink and Inventory Adjustments related metrics, which remain editable and can still be planned normally.

  • Retrend metrics: The fields for Retrend metrics populate only after a user selects the Retrend option in the Process Queue section of the main planning view toolbar.
  • Supersession metrics: The mapped metrics are: Stock On Hand (EOP -> BOP) and Actualized Sales Units. Actualized metrics display in the Sales Plan View. EOP values populate new item BOP values. Additionally,

    • Supersession date ranges are respected, even if they coincide oddly with phase dates. Specifically, supersession dates are used to narrow phase dates, but if no dates are set on the supersession, then phase dates are used.

    • If supersessions are set up for only some of the Inventories that make up a variant, then planned values are split using size curve and historical site contribution data.

  • System: Denoted by the word System at the beginning (for example: System Sales) and green row coloration. System metrics are those based on the values shown in the main planning and Cluster Sets views.
  • User: User metrics default to the system metrics of like name, but these are based on edits made in the Sales Plan view to the editable metrics. Thus, the user plan can diverge from the system plan in Sales Plan view.

LY metrics

LY metrics are only visible in the Receipt Plan view. LY metrics are not visible in the Sales Plan view.

Ship from Store (SFS) metric notes

Although Ship from Store values (Retail, Cost, Units) are reductions or negatives to e-commerce stock, it's more intuitive to plan SFS as a positive percentage. This is then used as a stock reduction through subtraction, rather than requiring a negative SFS percentage value.

There are no Product Hierarchy Defaults for the Ship from Store (SFS) metrics. Therefore, there will be no default SFS percent value which can be assigned to a higher product level. The SFS percent is managed at the Item level.

The metrics for Total Sales are not changed by the Ship from Store metrics as they display the Total Sales originating from that cluster set/cluster, regardless of fulfillment location. For example, e-commerce Total Sales will include all sales originating from e-commerce, including those fulfilled from stores. Retail Total Sales will consist of sales originating from Retail, NOT including SFS sales.

Also, ROS metrics are not changed. The ROS for the channel represents the ROS for Sales originating from that cluster set/cluster, regardless of fulfillment location.

Selling Period checkbox

Users can select the Selling Period checkbox to specify periods where sales are not planned for a selected placeholder or collection but is still in phase. If unchecked, then zeros out System Sales, System Sales Units, Sales, and Sales Units for the period. Thus, metrics in the main planning view and Cluster Set view are updated by this change. Users cannot enter sales for unchecked periods. When unchecked:

  • inventory (BOP, EOP) still calculates normally for the period
  • receipts can still be realized for the period
  • all Inventory Adjustments metrics (see below) can still be planned normally

Common uses for this feature might be items that repeat seasonally, such as holiday themed goods, where inventory is retained between selling periods.
