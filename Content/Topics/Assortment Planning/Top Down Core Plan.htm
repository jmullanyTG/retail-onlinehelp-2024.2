Top down core planning

Planners can view and manage sales and receipts in aggregate using the Top Down Core Plan view. The Top Down Core Plan view allows for simultaneously managing all placeholders for multiple collections instead of one collection or one placeholder at a time. Planners avoid the onerous and time consuming task of manually editing numerous individual collections or placeholders making sales and receipt planning much more efficient.

For example, planners can use the Multiple Changes right-click menu option to enter receipt quantities that are the same across multiple collections saving a great deal of clicking and saving.

Updates at lower levels aggregate up to higher levels while updates at higher levels split down proportionately to lower levels.

Comparison options

The comparison options are selected as a parameter.

Impact of making changes to plans in the Top Down Core Plan view

Changes made in the Top Down Core Plan view are seen as overrides in the main planning view and Sales Plan view.

Recalculating modified placeholders

There are editable Receipt metrics in the TDCP view. Any placeholders that have been modified have a status of Dirty and are flagged for Recalculation.

To recalculate modified placeholders, click the Recalculate Receipts button. Clicking this button initiates the same logic as the Recalculate Receipts button in the main planning view. This button uses the parameters defined within TDCP to calculate which placeholders to recalculate.

It is possible that multiple users might be affecting the same placeholders through different views (Receipt Plan view and TDCP), and also possible that these users might recalculate at similar times. To efficiently address this possibility, queue logic skips placeholders which are no longer dirty though the first recalculation. The recalculation does not auto-run the parameters as users could be working at the time.

Notifications are sent to users to indicate the status of the run. Clicking on the Notification takes users back to the TDCP.

Receipt Constraints are still managed exclusively through the Receipt Plan view, which supports multi-editing.

Recalculating receipt plans

Placeholders that have been modified have a receipt status of Dirty and are flagged for recalculation. To recalculate modified placeholders, click the Recalculate Receipts button. Recalculate Receipts uses the parameters defined within TDCP to determine which placeholders to recalculate.

The Recalculate Receipts functionality initiates the same logic as the Recalculate Receipts button in the main planning view.

It is possible that multiple users might be affecting the same placeholders through different views (Receipt Plan view and TDCP), and also possible that these users might recalculate at similar times. To efficiently address this possibility, queue logic skips placeholders which are no longer dirty though the first recalculation. The recalculation does not auto-run the parameters as users could be working at the time.

Notifications are sent to users to indicate the status of the run. Clicking on the Notification takes users back to the TDCP.

Receipt Constraints are managed exclusively through the Receipt Plan view, which supports multi-editing.

Snapshot data in the Comparison View

All snapshot data, including actualized and on order data, is included in the comparison. If a snapshot is compared to the plan immediately after creation, results are identical. If a snapshot is used for comparison some time after it has been taken, then the actualized periods as of the snapshot date are shown, and the future-planned periods as of the snapshot date are shown.

Cell locking

Locking is done at individual cell level. Only editable cells can be locked.

When a lock is in place, the entire row is colored to indicate lock status overriding any other formatting in place. When the lock is removed, the original formatting returns. Locks in the Top Down Core Plan view do not carry over into other views. Locking a Sales metric does not lock a Receipt metric and vice versa.

Parameters

The following parameters are available:

Parameter Description
Product Selects a product filter for the current view.
Site Hierarchy Selects the levels of the current Plan’s Site Hierarchy to include in the current view.
Cluster Sets Selects the cluster sets to include in the current view. The cluster sets listed include all default and "additionally available" cluster sets for the product level selected in the Product parameter.
Window of Time Returns the entire range between the minimum and maximum date selection and not just specified values. This is due to the BOP and STS rolling logic.

For example, if week 3 of August, weeks 2 and 3 of September, and all of November are selected, then August to November is returned in its entirety.

Otherwise, the values for August and September on the totals level would be different than the values for the selected weeks and may cause confusion.

Display Currency Sets the currency to display for all revenue based fields in the view. Defaults to the Home Currency for the active Assortment Plan.
Comparison Type The following options are available:
  • None
  • System Data
  • User Data
  • Snapshot options
Comparison Time Comparison types None, System Data, and User Data have the full set of dates mirroring the Window of Time parameter.

Snapshots have the dates specified within the snapshot only. This option behaves like the Window of Time parameter where selecting a few weeks may bring back extra rows because of the BOP and STS rolling logic.

For additional information, refer to Placeholder and collection view parameters.

Metrics notes

EOP and BOP values include Inter-Branch Transfers (IBT) for sites-in-cluster. Therefore, EOP and BOP values in all views and reports which use these metrics (such as the Sales Plan view and the Receipt Plan view) reflect values inclusive of IBT.

Field Description
Average Sales Units Average sales calculated by dividing Sales Units by Site Count.
AUR Always locked and shaded gray to indicate it's read-only.

The AUR grand total is a total for all months or across the time parameters selected. The calculation is: Grand Total AUR = Grand Total Sales / Grand Total Sales Units
Home AUT Weighted average (excludes zeros) unit ticket price for the period in Home Site currency.

Home AUT is editable in the main view and follows the same coloring rules as the other editable fields (Sales and Sales Units). Home AUT in the Comparison view is read-only.
Blended AUT Weighted average (excludes zeros) unit ticket price for the period, blended for all currencies present in the cluster. Blended AUT is derived from Home AUT, and like AUR is read-only.
AUT Discount % Impacts how AUR is derived from Blended AUT. The percent discount is taken first, and then the rest of the calculations proceed from there, namely applying VAT rates, price exceptions, and promotions. This metric averages across time and is editable in future periods.

It is important to note that AUT Discount % is applied and then any planned promotions are also applied. AUT Discount % is NOT inclusive of explicitly planned promotions.
BOP Units Beginning of Period (BOP) inventory units aggregated to the levels of the product and site hierarchies defined in assortment planning and merchandise financial planning.
Last Receipt Run The last date when a placeholder was processed through the Engine to calculate Receipts. A similar metric appears in the main planning view.
Receipt AUC Sets the average cost per unit for receipts in the period.
Receipt Cost Planned or actual cost of receipts for the period. The calculation is Receipt Units * Receipt AUC.
Receipt Period An editable checkbox in the current pane and read-only in the Comparison View pane. Changes to the checkbox in the current pane are fed through to the Receipt Plan view.
Receipt Status The calculation status of the receipt plan for the placeholder/collection (Clean or Dirty)
Receipt Units Editable. Sets the periods in which receipts can occur. Updates at lower levels aggregate up to higher levels while updates at higher levels split down proportionately to lower levels.

When System Data is selected as the Comparison Type, actualized receipts display in historical time periods and system receipt units in future time periods.

When a version of the plan is selected as the Comparison Type, the Receipt Units display that were in the plan for that specific snapshot

Changes override system Sales Units and Receipt Units and flow through to the following:

  • Assortment Rollup report

  • Reconciliation report

  • both Placeholder & Collection Receipt Plan views

    Changes made to the Sales Units and Receipt Units metric in these views flow through to the TDCP, read-only reports, and MFP Comparison

  • MFP comparison metric for Receipt Units: AP Primary Plan
Sales, Sales Units When Sales changes Sales Units updates and vice versa.

When changes to Sales or Sales Units are made to subtotals, totals, and grand totals, then data is spread proportionately and applicable PH sales Sales and Sales Units adjust.
Supplier Order (AUC, Cost, Units) Read-only. Sums at higher aggregations.

Calculation: Committed + Confirmed + Unknown Units On Order (Total Orders - Inter-Branch Transfers)
Supplier Order Plus Receipt (AUC, Cost, Units) Read-only. Sums at higher aggregations.

Calculation: Committed + Confirmed + Unknown + Receipt Units
Site Count A placeholder's active site count over time, based on the cluster set / clusters selected.
Stock To Sales This field updates in relation to sales changes and is flagged when above or below default thresholds.
