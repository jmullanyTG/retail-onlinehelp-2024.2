Order manager concepts

Order documents and receipt dates

To meet the requirements of the majority of specialty retailers, there is functionality for the determination of size and prepack breakdowns for orders. Due to the inherent inaccuracies associated with doing these analyses for multiple receipt dates simultaneously, order manager documents can only be created with a single receipt date each. Unlimited order lines, including different variants and store cluster orders, can be included as long as they all share a single receipt date.

This constraint also facilitates the tracking of outstanding activities for orders through the Upcoming Commitments Dashboard in Assortment Planning. For more information, see Assortment Upcoming Commitments Report.For more information, refer to the Assortment Planning User Guide.

Order workflow

Order Manager has been integrated with the Workflow module to facilitate identification of and movement through the various stages of the ordering process, as defined by the needs of the organization. Multiple workflows, ranging from very simple expedited flows, where all stages are completed at once with a single submission of the order, to very complex, with several levels of approval at each stage and submissions to external systems after each, can be configured and used within Order Manager. The number of days out from the receipt date that each stage must be completed is configured in the Workflow module and then compared to the current date in the above referenced Upcoming Commitments Dashboard, allowing users to track their progress against established deliverables.

For more information, see Workflow Manager overview. For more information, refer to the Workflow Manager User Guide.

Workflow status is set at the order line level of order documents, however, all lines in a document must have the same status to advance the document status and submit the order document to external systems. This is mainly relevant in pre-pack stage, as order lines can be selected for prepacking individually or in groups, in addition to all order lines at once

Status types in Order Manager

There are two types of status within Order Manager: document status and workflow status.

Document status describes the current ownership or state of the order document as a whole. Document statuses include things like Forwarded, Authorized, and Uploaded, and cannot be changed without customization.

Workflow status describes the stage of the active workflow that the order document is currently in. Workflow statuses are configurable through the Workflow module and can include anything an organization deems important. Available workflow statuses are dependent on the workflow currently assigned to an order document

Future recommended order horizons

When generating future recommended orders, ToolsGroup JustEnough extends the horizon specified in the Days From Today field to ensure that the order being placed has sufficient quantities to meet any inventory needs that may arise just beyond the order window provided by the user. This is especially critical for orders generated with short windows, such within 90 days.

The extended order horizon is calculated automatically based on the system policy Future Inventory Window which has a default value of 14 days. The calculation used to determine the effective horizon is (Days From Today value + supply chain lead time + Future Inventory Window value).

For supplier orders, the supply chain lead time is the maximum of the sum of lead times from the supplier all the way to the store. This is only done along the primary source path.

This extended horizon is only used for future recommended orders for “Orders Placed”. It is not used for future recommended orders for "Orders Delivered".

Reconsolidation orders

A reconsolidation order is executed based on a filter of items and their associated status. After the desired inventories are filtered and selected, the quantities to reconsolidate orders are driven by the source stock selection options.

Any inventories that are resolved to be sourced from a supplier will not be returned in a reconsolidation order.

The reconsolidation order generation process requires a supply chain to determine applicable stock transfer links, as well as a supply chain to determine which statuses are valid to the run. The parameter @iSupplyChainIDForStatusGathering will be set to the default supply chain defined using system policy Default Supply Chain (General: Planning) for all reconsolidation orders run through the UI. The parameter @iSupplyChainIDForReconsolidation is the specified supply chain in the Supply Chain field on the Reconsolidation Order main screen.

Replenish Engine Runs

Because reconsolidation is an entirely SQL based process, there is not an individual replenish engine run executed when an order is actioned. The stock status values used are based on the most recent engine run with the system policy defined default supply chain.
