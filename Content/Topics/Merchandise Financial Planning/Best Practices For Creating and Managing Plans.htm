Best practices for creating and managing plans

Here are some best practice suggestions to help get the most out of the available functionality in the plan creation and plan management sections of this module.

Master and branch plan set up tips

  • Master plans should contain all necessary product and location hierarchical components to properly manage an open-to-buy process.
  • Stay at a level that is reasonable in terms of the layers available to plan.  When managing an open-to-buy think of the spending on receipts and how to manage that spend at an executive type level.  Executives don’t want information that is too granular, so keep it at a level that is effective and everyone in the organization can understand.  An example would be planning at a category level or layer below division.  If total Female was a division and underneath that layer is categories of product such as denim, skirts, dresses, blouses, etc.  It’s very easy to see where the growth and declines of the business are occurring at that level.  Spend the time managing the open-to-buy at that level to move inventory investment into the right categories at the right time.
  • Create a master working branch underneath the master plan.  A master branch contains everything in the hierarchical structure for product and location.  This becomes the branch that everyone moves their changes into.  This allows for a holistic review of the business prior to approving a plan to merge into the master.  Users can then create their own branches off the master branch and merge their changes into the master branch.  Ensure the end date of this branch goes out as far as possible into the future.
  • When setting up read and write access for users, consider how much data they should see versus how much data they can edit.  An example is giving the Women’s denim planner write access for that category, but read access for total women’s.  They can make changes to Denim and understand what is happening in total Women’s.
  • When creating branches to edit the plans and there is a formal plan approval process in place, branch off the time periods that should be reviewed and approved.  This streamlines the process and is very helpful when conducting in-season plan reviews.  Additionally, upon approval only the changes to those time periods merge to the parent branch or master plan.

Snapshot utilization

Utilize snapshots as part of the plan review and approval process.  When a plan has been approved, take a snapshot so that there is record of the plans at that point in time.  This helps users understand Plan versus Actual and the variance between the two.

Use snapshot templates to schedule regular snapshots

  • Snapshot templates can be created to automatically take snapshots on a regular schedule. Schedule snapshots to be taken of the Master plan and Master working branch every week.  This makes reviewing the business simpler as the comparisons of plan vs. actual on a weekly basis are available directly in the main planning view.
  • Set an appropriate calendar level: The calendar level should be week if week to week planning is an important part of the business.  Otherwise, select Month as the calendar level. 

Create user specific master and branch plans

Be aware of the intended users when creating branch and sub-branch plans. Creating branches that only contain the product/site levels each user is responsible for speeds the planning process by limiting the amount of data the system has to read and process for each transaction. If the organization includes planners that are specifically concerned with higher levels of the product/site hierarchies, create additional Master plans for them that don’t include the lower levels of the hierarchy. These plans can ultimately be reconciled with lower level plans via Rebalance functionality.

Adding new leaf levels to existing plan hierarchies

As the business evolves, add new product/site levels to existing plans. Once the new leaf levels are created within the desired ItemMaster/Site Groups, first use the Initialize functionality on the Master plan, selecting ONLY THE NEW LEVEL(s) to be added.

Once complete, use the Reseed functionality on the branch plans beneath that master, again selecting ONLY THE NEW LEVEL(s) that were added to the master. In this manner, new levels can be set up in existing plans without losing the data for the existing levels.

Leaf level refers to an additional entity within an existing ItemMaster/Site Group.

For example, SiteGroup 4 is Site and contains 3 sites. If another site is added as a new site that is opening, then this is adding a new leaf level to SiteGroup 4.

This example does not pertain to adding a new structure level. An example of adding a new structure level would be when a Site Hierarchy consists of SiteGroups 1-4: Company, Country, District, Site and then a new SiteGroup, Region, is inserted between Country and District. This type of change constitutes a hierarchy restructure and requires that a new Master plan be created.
