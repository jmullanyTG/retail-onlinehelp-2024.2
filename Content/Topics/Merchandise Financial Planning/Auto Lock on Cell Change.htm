Automatically locking changed cells

When Auto Lock On Cell Change is enabled, changing the value of a sales-related metric automatically locks the cell (and any related cells). This saves having to stop and click Lock after each change.

  • The Auto Lock on Cell Change option follows the same behavior as the regular lock and spread functionality. It just replaces the need to click Lock after making cell changes. For more information, see Locking sales and dependent metrics.
  • Locks added automatically by the Auto Lock on Cell Change option can be removed in the same manner as manually added locks.
  • Auto Lock on Cell Change does not work on changes made through the Multiple Changes action. For more information, see Changing multiple cells at once.

Turning auto lock on cell change on or off

The Auto Lock On Cell Change option is located in the Calculation Toggles pane. Click the checkbox to toggle between enabling and disabling the option.

Example

This example illustrates the use of auto-lock when changing planned Sales values for the month of December.

  1. The Auto Lock On Cell Change option is enabled.

  2. The planned aggregate Sales value is changed for the month of December (the 2018 m_12 Total column).

    The new value for the month is $12,000,000 (see yellow box in the screenshot below).


    After making the change, the application automatically (see screenshot below):

    • updates the planned Sales Units value for 2018 m_12 Total from 111,506 to 142,381 to reflect the month's new planned Sales value.

    • locks the Sales, Sales Units, and AUR values for 2018 m_12 with the cells now formatted with a light red gradient background to indicate they are locked aggregated values.

  3. The Sales values are changed for each of the first 3 weeks in December (2018_wk48, 2018_wk49, and 2018_wk50) to $2,000,000 each.

    These changes are shown in the yellow boxes in the screenshot below.

    After making each change to a weekly Sales value, the system automatically:

    • updates the week's Sales Units value to reflect its new Sales value.
    • locks the Sales, Sales Units, and AUR values for that week with the cells are now formatted with a light red background to indicate their values are locked.

    • updates the Sales and Sales Units values of the remaining unlocked weeks to make up the difference between the newly-locked weekly values and the locked monthly value.

      In the screenshot above, the Sales values in the three locked weeks now account for $6,000,000 of the locked $12,000,000 monthly Sales value. The remaining $6,000,000 in Sales is now spread proportionally across the remaining unlocked weeks (2018_wk51 and 2018_wk52, marked in a green box in the screenshot above).

  4. The Sales value is changed for the 4th week in December (2018_wk51) to also be $2,000,000.

    After making this change, the system automatically:

    • updates the Sales Units value for 2018_wk51 to reflect the week's new Sales value.
    • locks the Sales, Sales Units, and AUR values for 2018_wk51.

    • updates the Sales and Sales Units value of the remaining unlocked week (2018_wk52, shown in the green box above) to make up the difference between the locked weekly values and the locked monthly values.

      In this example, the locked monthly Sales value is $12,000,000 and the combined Sales values of the locked weeks (weeks 48-51) is $8,000,000. This means the Sales value for week 52 has to be $4,000,000 for all the weeks in the month to aggregate to $12,000,000.

    • locks the Sales, Sales Units, and AUR values of the remaining unlocked week (2018_wk52) as a result of all the other periods in the month being locked AND the monthly value being locked. This lock is not a direct result of the Auto Lock on Change option as no change was manually entered in 2018_wk52.

      The lock is added because, logically, any changes to wk52 would require subsequent changes to either the aggregated monthly values (which is prevented by the locks to the monthly values added in step 2 above) or to the values in the other weeks in the month (which is prevented by the locks added in steps 3 and 4). Since wk52's values cannot be changed due to those other locks, the system automatically locks its values as well.
