Channels

With the introduction of Channels, it is imperative clients update their Strategy Process if they do not use omni channel strategy lines.

Coordinating channel relationships through the engine

The solution does not coordinate channel relationships through the engine. It is imperative that this is resolved with the CLIENT’S PROCESS.

For example, Strategy One - Retail, Strategy Two - Outlet has shared DCs and/or SDCs. The last strategy to run the engine will remove any OVERLAP inventory from the related strategies. So if Retail ran first and had 200 units on order for a related DC, but then Outlet ran second it would remove the 200 units from that related DC which can cause the entire order for Retail to not be realistic.

The client’s process must be that they run their highest priority channel first through the engine, then set them to a state that sets Pending Stock (i.e. pre-authorized). Then they can run the next priority channel, then set the Order Manager State. This process will be repeated until their channel priority list is exhausted.

Assortment Planning

For Placeholders with a defined markdown date, a regular ROS is calculated for the period from the phase-in date to the beginning of the markdown period. A markdown ROS is calculated for the period from the start of the markdown to the phase-out date.

Previously, if the markdown date was extended, then both the regular ROS for the new extended regular period and the markdown ROS for the new markdown period were recalculated. Importantly, the new regular ROS and markdown ROS were lower than before the markdown extension. These new lower rates of sale were backdated changing the sales plan for the defined period.

This changing of the regular ROS and the markdown ROS after the extension of a markdown date has been eliminated for v2020.1.

Now, if the markdown date is extended, then both the original regular ROS and original markdown ROS hold and apply to the new extended regular period and the new markdown period. Whatever the regular ROS was by week by number of weeks is applied to the new weeks that used to be markdown and are now regular.

The way markdowns are treated in Assortment Planning is fundamentally different now. It used to be just a date - nothing in the sales plan or receipt plan would change in response to setting a markdown.

Old functionality

  • assume you have a 10 week plan, with 100 sales, and a flat seasonality curve. The sales curve generated would then be: 10 / 10 / 10 / 10 / 10 / 10 / 10 / 10 / 10 / 10.
  • If you added a markdown date of Week 6, the sales plan would stay identical. If you added the markdown at Week 8, the sales plan would be identical.

As of ToolsGroup JustEnough v2019.3:

New functionality:

  • If you have no markdown date, the sales curve is the same as before: 10 units per week across the 10 weeks.
  • If you add a markdown date at Week 6, then the sales curve trends to 0 starting from week 6. The sales plan would be: 10 / 10 / 10 / 10 / 10 / 10 / 8 / 6 / 4 / 2, for a total sales value of 80.
  • If you add a markdown date at Week 8, then the sales plan would be: 10 / 10 / 10 / 10 / 10 / 10 / 10 / 10 / 6.7 / 3.3, for a total sales value of 90.

Thus, because the Markdown Date now influences the generated sales plan (which, in turn, influences the generated receipt plan), then all the other existing ways to change a sales plan now interact with this new feature. Changing phase dates affects the sales plan - the way in which it affects the sales plan depends on whether a markdown dates is set, and where in the plan it is set. Changing total sales will split proportionately across the pre-markdown and post-markdown halves of the sales plan. Adding and removing a markdown date will change your sales total. Adding a markdown date to a new placeholder before adding a sales total will behave differently than adding a sales total before adding a markdown date.

The system no longer taking the markdown portion of a plan into account

Historically, the system took the demand from the markdown portion of a plan into account and recommended a total matching the Sales plan which not is desired from a stock holding position. As of v2019.3, when the new system policy labeled Allow Receipts After Markdown (Maintain > Policies > Engine: Commitments) is set to False (default), the Receipt plan only considers the regular portion of the Sales plan and ignores the markdown phase of the plan. Set the system policy to True if additional stock should be ordered beyond the planned markdown date.

Revisions to the Sell Through metric

The metric, Sell Through, was user-editable and changes affect the metric Additional Buy. The Sell Through metric is now read-only. Additionally, Sell Through used to represent the ratio between total sales and total buy. Now this metric represents total sales/total orders.

On Order now included in all EOP/BOP values

Previously, On Order (the sum of Committed, On Order, and On Order (Unknown) was not included in Reconcile and Rollup reports. As of v2019.3, On Order is included in all EOP/BOP values. Therefore, customers who customized the formula to include On Order should verify that On Order is not double counted.

Selecting and applying LifeCycle Curves

The new Lifecycle Curve allows two sets of logic to apply to a sales plan simultaneously. In the existing Seasonality Curve concept, a curve can be chosen to generate a sales plan that reflects the expected seasonal influences over time, like coats peaking in winter.

For long-selling items, the Seasonality Curve will allow for the same seasonal effects to repeat over the years. The addition of the Lifecycle curve allows for the expected trends over the life of the item to also be taken into account - for example, the item selling strongly at the beginning of its life, and tapering down over time.

The Lifecycle Curve can be set for placeholders in the new field Lifecycle Curve in the Placeholders and Collections views, and they can be set for specific cluster set / clusters in the Cluster Sets view.

Weighting the Curves

There is a weighting of the Seasonality Curve and the Life Cycle Curve. This weighting takes into account the fact that a product has a certain life cycle with a particular Curve but you are now using a non-comparable period. You may want X% of the Curve based on the Seasonality Curve and Y% based on the Life Cycle Curve. Once you have determined the percentages you add the value to a new field, Seasonality Curve Factor.

Seasonality Curve Factor

When an Item has both a Seasonality Curve and a Life Cycle Curve, the Seasonality Curve Factor determines how the two curves influence the Sales Plan. A value of .5 means that the two curves each influence the Sales Plan 50% each. A value of .7 would mean that the Seasonality Curve is weighted at 70% and the Life Cycle Curve at 30%. The values for the two curves always add up to 100%.

The Seasonality Curve Factor is only relevant if there is both a Seasonality Curve and a Life Cycle Curve. If only one curve is set, then any value in the Seasonality Curve Factor field has no effect.

Site Counts

The number of sites across the life span of the non-comparable period can be taken into consideration. For example, the initial life cycle may only have had 4 sites but the new life cycle has 6 sites. As a result, Sales need to respond to the difference in Site Count and accommodate for the additional sales at the additional sites.

Whether or not site counts are included in adjusting Sales Plans is controlled by existing system policy Toggle on/off snapshot aware seasonal index calculation. If set to True, site counts by period will be included in adjusting Sales Plans and, therefore, the Sales Plan.

Supersessions in AP: Using an existing variant's EOP as the BOP for a subsequent variant

v2021.1 introduces existing Supersession functionality into AP. Supersessions let users forecast demand for new items with no sales history by using an existing variant's EOP as the BOP for the new item.

Supersession relationships are defined in the Supersession solution at the Item Master level (style-color-size) and best practice is to map all sizes to create a 1-to-1 relationship. For multi-size variants, Supersessions are defined by size, for each size. New Supersessions defined in the Supersession solution and changes to existing Supersessions are automatically pushed into AP.

Typically, outside of the new Supersession functionality AP is at the Variant level (style-color) and does not consider size.

Supersession functionality in AP is enabled by setting new system policy Enable Supersessions to True. For more information, see System Policies.

Example scenario

There are times when a new variant is close enough to an existing variant that the two variants are seen by customers as interchangeable. (For example, Variant A = red socks with black squares, Variant B = red socks with black circles.)

In such a scenario, the stock on-hand position of the first can be considered when planning stock needs for the second variant. This is very useful when:

  • both items look very similar

  • Variant B’s lifecycle is after Variant A’s lifecycle and the user would like to use Variant A’s EOP as Variant B’s BOP

    Phase dates need to be considered when utilizing supersessions.

  • the supplier of an existing variant produces the same style each year, but changes the style-code/SKU in subsequent years
  • when stock from one item should be considered towards stock for a subsequent item.

    2020.4

    Forecast Level Optimization

    Forecast Level Optimization is a new tool designed to help users optimize the accuracy of their forecasts. This is valuable for demand planners who have highly intermittent, short life cycle sales patterns. The purpose of this tool is to generate and automatically test a variety of leveling schemes, and to select the one that produces the most accurate inventory-level forecasts. Forecasting such series manually can be very inaccurate and difficult. To combat this problem, level optimization groups inventories into levels based on a hierarchy. These levels ideally have consistent sale behavior and therefore, can be forecasted accurately. Level forecasts are then split down to inventory.

    There are certain requirements needed in order to successfully run the forecast level optimization. Users should work with their ToolsGroup JustEnough representative to properly configure their database instances, and ensure the correct inputs are set up for accurate results.

    Modeling sales data

    If sales are reduced significantly due to a large scale event such as a disaster, you can put in place modeled sales using the Disaster Sales Modeling screen. You can select a store and have the system calculate relevant theoretical sales history data for this store during the periods impacted by the event.

    The modeled sales history data for selected stores is based on forecasting methods. The data is modeled on the LY periods for the same store at the same levels of the product hierarchies. This data is entered as a special sales type so that it does not conflate or otherwise interfere with actual sales.
